# Fabric EC2 — 콘솔 · CLI 완전 따라하기 (SSM 접속)

> **목표:** AWS EC2 1대 생성 → Hyperledger Fabric + Anchor Gateway 기동 → EKS Backend 연결  
> **리전:** `ap-northeast-2` (서울) · **프로젝트:** `forenshield`  
> **EC2 접속:** **SSM Session Manager** (SSH 22 · `.pem` · 퍼블릭 IP **불필요**)  
> **코드:** [`Infra/fabric/`](../fabric/)  
> **BE:** 이미 구현 완료 (`HttpBlockchainAnchorClient`)

## 1. 전체 그림

```text
┌─────────────────────────────────────────────────────────────┐
│  VPC forenshield (10.0.0.0/16) — EKS와 동일                  │
│                                                              │
│  [팀원 PC] ──SSM Session Manager──► [EC2 fabric-poc]        │
│       (SSH 22 없음, IAM 권한만)      private subnet          │
│                                                              │
│  [EKS backend Pod] ──8088──► [EC2 Gateway :8088]            │
│         │                         ├─ Docker: Fabric peer     │
│         │ VPN                     └─ Node Gateway            │
│         ▼                                                    │
│  [On-Prem GPU :8000]  ← AI만 (기존, 변경 없음)               │
└─────────────────────────────────────────────────────────────┘
```

| 단계 | 소요(대략) | 결과 |
|------|------------|------|
| IAM + SG + EC2 | 15~25분 | Private EC2 (SSM 등록) |
| `setup-all.sh` | 30~60분 | Fabric + chaincode |
| Gateway + EKS | 10분 | BE 앵커 동작 |

**SSM을 쓰는 이유:** SSH 22를 열지 않아도 되고, **팀원마다 IP를 SG에 추가할 필요 없음**. AWS IAM 권한이 있는 팀원은 누구나 동일하게 EC2 셸 접속 가능.

## 2. 사전 확인

### 2.1 필요한 것

| 항목 | 설명 |
|------|------|
| AWS 계정 | ForenShield 운영 계정 (`forenshield` 프로필) |
| VPC | **이미 EKS가 올라가 있는 VPC** (새 VPC 만들지 않음) |
| NAT Gateway | Private subnet EC2가 SSM·`git clone` 하려면 **아웃바운드 인터넷** 필요 (ForenShield VPC에 이미 있음) |
| EKS 노드 SG | `forenshield-sg-eks-node` → Gateway **8088** 인바운드 소스 |
| Git | EC2에서 Infra 레포 `git clone` (또는 팀 repo URL) |
| 팀원 IAM | `ssm:StartSession` 등 SSM 접속 권한 (콘솔 PowerUser 이상이면 보통 OK) |

> **키 페어(`.pem`)는 필요 없습니다.** 이 가이드에서는 생성·연결하지 않습니다.

### 2.2 CLI로 SSM 쓸 팀원 (선택)

로컬 PC에서 `aws ssm start-session`을 쓰려면 **Session Manager Plugin** 설치:

- [AWS Session Manager Plugin 설치](https://docs.aws.amazon.com/systems-manager/latest/userguide/session-manager-working-with-install-plugin.html)
- 콘솔 **「연결 → Session Manager」** 만 쓸 경우 플러그인 불필요

### 2.3 기존 리소스 ID 찾기

**Bash / Git Bash / WSL:**

```bash
export AWS_PROFILE=forenshield
export AWS_REGION=ap-northeast-2

# VPC
export VPC_ID=$(aws eks describe-cluster --name forenshield \
  --query 'cluster.resourcesVpcConfig.vpcId' --output text)
echo "VPC_ID=$VPC_ID"

# EKS 노드 SG (※ cluster SG 가 아님!)
aws ec2 describe-security-groups --filters "Name=vpc-id,Values=$VPC_ID" \
  --query "SecurityGroups[?contains(GroupName, 'forenshield') || contains(GroupName, 'eks')].[GroupId,GroupName]" \
  --output text
```

**`SG_EKS` = `forenshield-sg-eks-node`** (ForenShield 예: `sg-0f59d7bff68ff911f`)

> `eks-cluster-sg-forenshield-...` 는 **클러스터 SG**입니다. 8088 소스로 쓰지 마세요.

```bash
export SG_EKS=sg-0f59d7bff68ff911f   # 본인 환경의 forenshield-sg-eks-node

# Private subnet (EKS worker용 — data/public 제외)
aws ec2 describe-subnets --filters "Name=vpc-id,Values=$VPC_ID" \
  --query "Subnets[?Tags[?Key=='Name' && contains(Value, 'private') && !contains(Value, 'data')]].[SubnetId,Tags[?Key=='Name'].Value|[0],AvailabilityZone,CidrBlock]" \
  --output table
```

**`SUBNET_ID` = `forenshield-private-a` 또는 `forenshield-private-b` 중 하나**

ForenShield 예:

```bash
export SUBNET_ID=subnet-01b1fa0d84ae91e49   # forenshield-private-a, 10.0.10.0/24
# 또는 subnet-03952619c95bc612e              # forenshield-private-b, 10.0.11.0/24
```

**PowerShell:**

```powershell
$env:AWS_PROFILE = "forenshield"
$env:AWS_REGION = "ap-northeast-2"
$env:VPC_ID = aws eks describe-cluster --name forenshield `
  --query "cluster.resourcesVpcConfig.vpcId" --output text
aws ec2 describe-security-groups --filters "Name=vpc-id,Values=$env:VPC_ID" `
  --query "SecurityGroups[?contains(GroupName, 'forenshield')].[GroupId,GroupName]" --output text
```

메모해 둘 변수:

| 변수 | ForenShield 예시 | 용도 |
|------|------------------|------|
| `VPC_ID` | `vpc-0a074ca9d04307a9c` | SG·EC2 |
| `SG_EKS` | `sg-0f59d7bff68ff911f` | Gateway 8088 허용 소스 |
| `SUBNET_ID` | `subnet-01b1fa0d84ae91e49` | EC2 위치 (private) |

## 3. Step A — IAM 역할 (SSM용)

EC2가 SSM에 등록되려면 **인스턴스 프로필**에 `AmazonSSMManagedInstanceCore` 가 붙어 있어야 합니다.

### 3.1 AWS 콘솔

1. [IAM 콘솔](https://console.aws.amazon.com/iam/) → **역할** → **역할 생성**
2. **신뢰할 수 있는 엔터티:** AWS 서비스 → **EC2** → 다음
3. **권한 정책:** `AmazonSSMManagedInstanceCore` 검색 후 선택 → 다음
4. **역할 이름:** `forenshield-ec2-fabric-role` → **역할 생성**
5. 역할 상세 → **인스턴스 프로필 ARN** 메모 (또는 아래 CLI로 프로필 생성)

> 콘솔에서 역할만 만들면 인스턴스 프로필이 자동 생성되는 경우가 많습니다. EC2 시작 화면 **IAM 역할** 드롭다운에 `forenshield-ec2-fabric-role` 이 보이면 OK.

### 3.2 AWS CLI

```bash
export AWS_PROFILE=forenshield
export AWS_REGION=ap-northeast-2

# 신뢰 정책 (EC2가 역할 assume)
cat > /tmp/trust-ec2.json <<'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
EOF

aws iam create-role --role-name forenshield-ec2-fabric-role \
  --assume-role-policy-document file:///tmp/trust-ec2.json

aws iam attach-role-policy --role-name forenshield-ec2-fabric-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore

aws iam create-instance-profile --instance-profile-name forenshield-ec2-fabric-profile
aws iam add-role-to-instance-profile \
  --instance-profile-name forenshield-ec2-fabric-profile \
  --role-name forenshield-ec2-fabric-role

# IAM 전파 대기 (30~60초)
sleep 10
echo "INSTANCE_PROFILE=forenshield-ec2-fabric-profile"
```

**검증:**

```bash
aws iam get-instance-profile --instance-profile-name forenshield-ec2-fabric-profile \
  --query 'InstanceProfile.Roles[0].RoleName' --output text
# → forenshield-ec2-fabric-role
```

## 4. Step B — Security Group

Fabric Gateway **TCP 8088** — **EKS 노드 SG에서만** 허용.  
**SSH 22 규칙은 추가하지 않습니다** (SSM 사용).

### 4.1 AWS 콘솔

1. [EC2 콘솔](https://ap-northeast-2.console.aws.amazon.com/ec2/home?region=ap-northeast-2) → **보안 그룹** → **보안 그룹 생성**
2. 입력:
   - **이름:** `forenshield-sg-fabric`
   - **설명:** `ForenShield Fabric Anchor Gateway`
   - **VPC:** EKS와 동일 VPC
3. **인바운드 규칙** — **1개만** 추가:

   | 유형 | 포트 | 소스 | 설명 (영문·ASCII만) |
   |------|------|------|---------------------|
   | 사용자 지정 TCP | **8088** | **보안 그룹** → `forenshield-sg-eks-node` | `EKS to Gateway` |

> **설명 필드 주의:** 한글·`→` 같은 특수문자는 **불가**. 영문·숫자와 `._-:/()#,@[]+=&;{}!$*` 만 허용. 비우거나 `EKS to Gateway` 처럼 입력.

4. **아웃바운드:** 기본(전체 허용) 유지 → **생성**
5. SG ID 메모 → `SG_FABRIC`

> 8088을 `0.0.0.0/0`에 열지 마세요. SSH 22도 열지 마세요.

### 4.2 AWS CLI

```bash
export AWS_PROFILE=forenshield
export AWS_REGION=ap-northeast-2
export VPC_ID=vpc-xxxxxxxx
export SG_EKS=sg-xxxxxxxx              # forenshield-sg-eks-node

export SG_FABRIC=$(aws ec2 create-security-group \
  --group-name forenshield-sg-fabric \
  --description "ForenShield Fabric Anchor Gateway" \
  --vpc-id "$VPC_ID" \
  --query GroupId --output text)
echo "SG_FABRIC=$SG_FABRIC"

aws ec2 authorize-security-group-ingress --group-id "$SG_FABRIC" \
  --protocol tcp --port 8088 --source-group "$SG_EKS"
```

**검증:**

```bash
aws ec2 describe-security-groups --group-ids "$SG_FABRIC" \
  --query 'SecurityGroups[0].IpPermissions' --output table
```

## 5. Step C — EC2 인스턴스 생성

Private subnet · **퍼블릭 IP 없음** · **키 페어 없음** · **IAM 인스턴스 프로필 필수**.

### 5.1 AWS 콘솔

1. EC2 → **인스턴스** → **인스턴스 시작**
2. **이름:** `forenshield-fabric-poc`
3. **AMI:** Ubuntu Server 22.04 LTS (HVM), 64-bit (x86)  
   (Ubuntu 22.04는 **SSM Agent 기본 포함**)
4. **인스턴스 유형:** `t3.medium` (4 GiB RAM — 부족하면 `t3.large`)
5. **키 페어:** **「키 페어 없이 계속」** 선택
6. **네트워크 설정** → **편집:**
   - **VPC:** EKS VPC
   - **서브넷:** `forenshield-private-a` 또는 `forenshield-private-b`
   - **퍼블릭 IP 자동 할당:** **비활성화**
   - **보안 그룹:** `forenshield-sg-fabric`
7. **스토리지:** 30 GiB gp3
8. **고급 세부 정보** → **IAM 인스턴스 프로필:** `forenshield-ec2-fabric-role` (또는 `forenshield-ec2-fabric-profile`)
9. **인스턴스 시작**
10. 메모:
    - **인스턴스 ID** → `INSTANCE_ID`
    - **프라이빗 IPv4** → `EC2_PRIVATE_IP` (EKS `BLOCKCHAIN_ANCHOR_URL`에 사용)

### 5.2 AWS CLI

```bash
export AWS_PROFILE=forenshield
export AWS_REGION=ap-northeast-2

export AMI_ID=$(aws ec2 describe-images \
  --owners 099720109477 \
  --filters "Name=name,Values=ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*" \
            "Name=state,Values=available" \
  --query 'sort_by(Images,&CreationDate)[-1].ImageId' --output text)

export SUBNET_ID=subnet-xxxxxxxx
export SG_FABRIC=sg-xxxxxxxx

export INSTANCE_ID=$(aws ec2 run-instances \
  --image-id "$AMI_ID" \
  --instance-type t3.medium \
  --subnet-id "$SUBNET_ID" \
  --security-group-ids "$SG_FABRIC" \
  --iam-instance-profile Name=forenshield-ec2-fabric-profile \
  --no-associate-public-ip-address \
  --block-device-mappings '[{"DeviceName":"/dev/sda1","Ebs":{"VolumeSize":30,"VolumeType":"gp3","DeleteOnTermination":true}}]' \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=forenshield-fabric-poc},{Key=Project,Value=forenshield}]' \
  --query 'Instances[0].InstanceId' --output text)

echo "INSTANCE_ID=$INSTANCE_ID"
aws ec2 wait instance-running --instance-ids "$INSTANCE_ID"

export EC2_PRIVATE_IP=$(aws ec2 describe-instances --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[0].Instances[0].PrivateIpAddress' --output text)
echo "EC2_PRIVATE_IP=$EC2_PRIVATE_IP"
```

### 5.3 SSM 등록 대기

인스턴스 기동 후 **2~5분** 뒤 SSM에 등록됩니다.

```bash
aws ssm describe-instance-information \
  --filters "Key=InstanceIds,Values=$INSTANCE_ID" \
  --query 'InstanceInformationList[0].PingStatus' --output text
# → Online 이 될 때까지 반복
```

콘솔: EC2 → 인스턴스 선택 → **연결** 탭 → **Session Manager** 항목이 활성화될 때까지 대기.

## 6. Step D — SSM 접속

팀원 누구나 **동일한 방법**으로 접속 (본인 IP를 SG에 넣을 필요 없음).

### 6.1 AWS 콘솔 (권장)

1. EC2 → `forenshield-fabric-poc` 선택
2. **연결** → **Session Manager** 탭 → **연결**
3. 브라우저 터미널에서 `ubuntu` 셸이 열림

접속 확인:

```bash
whoami          # ubuntu
uname -a
free -h
df -h
```

### 6.2 AWS CLI

```bash
export AWS_PROFILE=forenshield
export AWS_REGION=ap-northeast-2
export INSTANCE_ID=i-xxxxxxxxxxxxxxxxx

aws ssm start-session --target "$INSTANCE_ID"
```

> 로컬에 Session Manager Plugin이 없으면 `SessionManagerPlugin is not found` 오류 → [2.2](#22-cli로-ssm-쓸-팀원-선택) 참고.

### 6.3 팀원 권한

| 역할 | 필요 권한 |
|------|-----------|
| EC2 셸 (SSM) | `ssm:StartSession`, `ssm:TerminateSession` (대상 인스턴스/태그) |
| E2E 검증만 | `kubectl` + AWS EKS — **SSM 불필요** |
| Fabric 로그 확인 | SSM 또는 담당자 공유 |

`install-ec2-prereqs.sh` 실행 후 docker 그룹 반영을 위해 **SSM 세션 종료 → 다시 연결** (SSH 재접속과 동일).

## 7. Step E — Fabric + Gateway 설치

## 8. Step F — EKS Backend 연결

### 8.1 ConfigMap 수정

파일: [`Infra/config/k8s/app-config.yaml`](../config/k8s/app-config.yaml)

```yaml
  BLOCKCHAIN_ANCHOR_ENABLED: "true"
  BLOCKCHAIN_ANCHOR_MODE: "http"
  BLOCKCHAIN_ANCHOR_URL: "http://<EC2_PRIVATE_IP>:8088/api/v1/anchor"
  BLOCKCHAIN_ANCHOR_NETWORK: "hyperledger-fabric-forenshield"
```

예: `http://10.0.10.45:8088/api/v1/anchor`

> **반드시 Private IP.**

### 8.2 적용

```bash
export AWS_PROFILE=forenshield
aws eks update-kubeconfig --name forenshield --region ap-northeast-2

kubectl apply -f Infra/config/k8s/app-config.yaml -n forenshield
kubectl rollout restart deployment/backend -n forenshield
kubectl rollout status deployment/backend -n forenshield
```

환경 변수 확인:

```bash
kubectl exec -n forenshield deploy/backend -- env | grep BLOCKCHAIN
```

## 10. 중지 · 비용 절감

### Fabric + Gateway만 중지

```bash
# SSM 세션 안에서
pkill -f "node src/server.js" || true
cd ~/forenshield-fabric-runtime/fabric-samples/test-network
./network.sh down
```

### EC2 중지

**콘솔:** EC2 → 인스턴스 → **중지**

**CLI:**

```bash
aws ec2 stop-instances --instance-ids "$INSTANCE_ID"
aws ec2 wait instance-stopped --instance-ids "$INSTANCE_ID"
```

재시작 후 Fabric은 `setup-all.sh` 재실행 또는 AMI 백업 전략 검토.

퇴근 루틴: [`14.daily-cost-shutdown.md`](./14.daily-cost-shutdown.md)

## 11. 트러블슈팅

| 증상 | 원인 | 조치 |
|------|------|------|
| SG 생성 시 `Invalid rule description` | 규칙 설명에 한글·`→` 등 | 설명을 `EKS to Gateway` 로 바꾸거나 **비우기** |
| SSM `Target not connected` | Agent 미등록, NAT 없음 | `PingStatus` 확인, NAT·아웃바운드 |
| `git clone` / `apt` 실패 | Private subnet NAT | VPC NAT Gateway·라우트 테이블 확인 |
| `setup-all.sh` docker error | docker 그룹 | `sudo usermod -aG docker ubuntu` → SSM 재연결 |
| chaincode deploy 실패 | 디스크·메모리 | `df -h`, `t3.large` |
| `curl localhost:8088` 실패 | Gateway 미기동 | `gateway.log`, `npm start` |
| EKS `Connection refused` | SG 8088, 잘못된 IP | **Private IP**, `SG_FABRIC` ← `forenshield-sg-eks-node` |
| BE `FAILED` | Gateway 502 | SSM으로 EC2에서 앵커 curl 재현 |
| `hashValid=false` | 파일 변경됨 | Fabric 문제 아님 — 해시 불일치 |
| CLI `SessionManagerPlugin not found` | 플러그인 미설치 | [2.2](#22-cli로-ssm-쓸-팀원-선택) 또는 콘솔 접속 |

### SSM이 계속 Offline일 때

```bash
# 인스턴스 프로필 확인
aws ec2 describe-instances --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[0].Instances[0].IamInstanceProfile'

# SSM Agent (SSM 세션이 되면)
sudo systemctl status snap.amazon-ssm-agent.amazon-ssm-agent.service
# 또는
sudo systemctl status amazon-ssm-agent
```

NAT 없이 SSM만 쓰려면 VPC Interface Endpoint 3개 필요: `com.amazonaws.ap-northeast-2.ssm`, `ssmmessages`, `ec2messages` (ForenShield는 NAT 있음).
