## 1. 도메인 접속 시 503 Service Temporarily Unavailable

### 증상

```bash
kubectl apply -f Infra/config/k8s/ingress.yaml
kubectl get ingress -n forenshield
# NAME                  CLASS   HOSTS                      ADDRESS   PORTS
# forenshield-ingress   alb     forensheildjangdochi.com             80     ← ADDRESS가 비어 있음!

curl https://forensheildjangdochi.com/health
# 503 Service Temporarily Unavailable
```

문제는 두 가지가 겹쳐 있었습니다.

1. **ADDRESS가 비어 있음** → ALB Controller가 ALB를 만들지 못하고 있음
2. **503** → 도메인이 Terraform으로 만든 **기존 ALB**를 가리키는데, 거긴 타겟이 하나도 없음

### 1-1. 원인: ALB Controller IAM 권한 부족

Ingress 이벤트를 확인하니 권한 에러가 반복되고 있었습니다.

```bash
kubectl describe ingress forenshield-ingress -n forenshield
# Warning  FailedBuildModel  ... UnauthorizedOperation:
# ... is not authorized to perform: ec2:DescribeRouteTables ...
```

ALB Controller가 서브넷을 자동 탐색하려면 `ec2:DescribeRouteTables` 권한이 필요한데,
클러스터에 설치된 `AWSLoadBalancerControllerIAMPolicy`가 **구버전이라 이 권한이 빠져** 있었습니다.

### 1-2. 해결: IAM 정책에 권한 추가 + 컨트롤러 재시작

```bash
# 1) 현재 정책 문서를 받아서 ec2:DescribeRouteTables 추가
ARN=arn:aws:iam::877044078824:policy/AWSLoadBalancerControllerIAMPolicy
VER=$(aws iam get-policy --policy-arn $ARN --query "Policy.DefaultVersionId" --output text)
aws iam get-policy-version --policy-arn $ARN --version-id $VER \
  --query "PolicyVersion.Document" --output json > policy-doc.json

# policy-doc.json 안의 "ec2:DescribeSubnets" 옆에 "ec2:DescribeRouteTables" 추가 후

# 2) 새 정책 버전(v2) 생성 + 기본 버전으로 설정
aws iam create-policy-version --policy-arn $ARN \
  --policy-document file://policy-doc.json --set-as-default

# 3) 컨트롤러 재시작 (즉시 재시도하게)
kubectl rollout restart deployment/aws-load-balancer-controller -n kube-system
kubectl rollout status deployment/aws-load-balancer-controller -n kube-system
```

→ 약 1~2분 후 새 ALB(`k8s-forenshi-forenshi-fabf0c2de3`)가 생성됨.

## 2. ALB 타겟이 전부 unhealthy (ResponseCodeMismatch)

### 증상

```bash
aws elbv2 describe-target-health --target-group-arn <TG_ARN>
# State: unhealthy, Reason: Target.ResponseCodeMismatch
```

### 원인

ALB 헬스체크 기본 경로가 `/`인데,

- **백엔드**: `/`는 404 (그런 엔드포인트 없음)
- **프론트엔드**: `/`는 307 리다이렉트 (Next.js가 `/main`으로 리다이렉트)

헬스체크는 200을 기대하므로 둘 다 실패.

### 해결: Service에 헬스체크 경로 어노테이션 추가

```yaml
# Infra/config/k8s/backend/service.yaml
metadata:
  name: backend
  namespace: forenshield
  annotations:
    alb.ingress.kubernetes.io/healthcheck-path: /actuator/health
```

```yaml
# Infra/config/k8s/frontend/service.yaml
metadata:
  name: frontend
  namespace: forenshield
  annotations:
    alb.ingress.kubernetes.io/healthcheck-path: /health   # nginx가 항상 200 반환
```

```bash
kubectl apply -f Infra/config/k8s/backend/service.yaml -f Infra/config/k8s/frontend/service.yaml
# 1~2분 후 전부 healthy 전환 확인
```

## 3. Route 53이 옛 ALB를 가리킴 (503의 직접 원인)

### 원인

`forensheildjangdochi.com` A 레코드가 Terraform으로 만든 기존 `forenshield-alb`를
가리키고 있었습니다. 트래픽을 실제로 받는 건 Ingress가 새로 만든 ALB이므로 레코드를 바꿔야 합니다.

### 해결: A 레코드를 새 ALB로 UPSERT

```bash
# 새 ALB의 DNS/Zone 확인
aws elbv2 describe-load-balancers --names k8s-forenshi-forenshi-fabf0c2de3 \
  --query "LoadBalancers[0].{DNS:DNSName,Zone:CanonicalHostedZoneId}"

# Route 53 레코드 변경 (Alias A 레코드 UPSERT)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z091888114TC5UULOEOX5 \
  --change-batch file://r53-change.json
```

### 검증

```bash
curl -s -o /dev/null -w "%{http_code}" -L https://forensheildjangdochi.com/   # 200
curl https://forensheildjangdochi.com/actuator/health                         # {"status":"UP"}
```

> 💡 기존 `forenshield-alb`는 이제 트래픽을 받지 않으므로, 비용 절감을 원하면 Terraform에서 정리해도 됩니다.

## 4. 파일 업로드 테스트 → S3 자격증명 에러

### 4-1. curl (26) Failed to open/read local data

```bash
curl -F "file=@test.mp4" https://forensheildjangdochi.com/api/evidences/upload
# curl: (26) Failed to open/read local data from file/application
```

**원인**: 현재 디렉토리에 `test.mp4`가 없어서. 서버 문제가 아니라 로컬 파일 경로 문제입니다.
파일명에 공백/괄호가 있으면 따옴표로 감싸고 **공백까지 정확히** 입력해야 합니다.

```bash
# 실제 파일명: aws_architecture.drawio (3).png  ← 괄호 앞 공백 주의!
curl -F "file=@/c/Users/user/Downloads/aws_architecture.drawio (3).png" \
  https://forensheildjangdochi.com/api/evidences/upload
```

### 4-2. FILE_UPLOAD_FAILED — AWS SDK `sts` 모듈 누락

```json
{"success":false,"errorCode":"FILE_UPLOAD_FAILED","message":"파일 업로드에 실패했습니다."}
```

Pod 로그 확인:

```bash
kubectl logs -n forenshield deployment/backend --since=10m | grep -A5 ERROR
# SdkClientException: Unable to load credentials from any of the providers ...
# WebIdentityTokenCredentialsProvider(): To use web identity tokens,
#   the 'sts' service module must be on the class path.   ← 핵심!
```

**원인**: IRSA 설정 자체는 정상 (Pod에 `AWS_ROLE_ARN`, `AWS_WEB_IDENTITY_TOKEN_FILE` 환경변수 주입 확인).
그런데 AWS SDK가 이 웹 아이덴티티 토큰을 IAM 자격증명으로 **교환하려면 `sts` 모듈이 필요**한데
`build.gradle`에 없었습니다.

**해결**: `build.gradle`에 의존성 추가 후 이미지 재배포.

```groovy
implementation platform('software.amazon.awssdk:bom:2.31.20')
implementation 'software.amazon.awssdk:s3'
// IRSA(웹 아이덴티티 토큰) 자격증명 교환에 필수
implementation 'software.amazon.awssdk:sts'
```

```bash
cd /c/Final_Project/backend/backend-forensic

docker build -t forenshield-backend .
aws ecr get-login-password --region ap-northeast-2 | \
  docker login --username AWS --password-stdin 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com
docker tag forenshield-backend:latest 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com/forenshield-backend:latest
docker push 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com/forenshield-backend:latest

kubectl rollout restart deployment/backend -n forenshield
kubectl rollout status deployment/backend -n forenshield
```

→ 재배포 후 업로드 성공, S3에 파일 저장 확인.

## 5. Argo CD 설치

### 5-1. 설치 — CRD가 너무 커서 실패

```bash
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
# The CustomResourceDefinition "applicationsets.argoproj.io" is invalid:
# metadata.annotations: Too long: must have at most 262144 bytes
```

**원인**: 일반(클라이언트 사이드) apply는 리소스 전체를 `last-applied-configuration`
어노테이션에 복사하는데, 이 CRD는 정의가 어노테이션 최대 크기(256KB)를 초과합니다.

**해결**: server-side apply 사용 (이 어노테이션을 만들지 않음).

```bash
kubectl apply --server-side -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

### 5-2. server-side apply 충돌 (field manager conflict)

```text
Apply failed with 1 conflict: conflict with "kubectl-client-side-apply" ...
```

**원인**: 첫 설치(client-side)가 만든 리소스의 필드 소유자가 달라서 생기는 충돌.
같은 매니페스트이므로 강제 덮어쓰기 하면 됩니다.

```bash
kubectl apply --server-side --force-conflicts -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

kubectl get pods -n argocd   # 7개 Pod 전부 Running 확인
```

### 5-3. UI 접속

```bash
# 초기 admin 비밀번호 확인
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d; echo

# 포트포워딩
kubectl port-forward svc/argocd-server -n argocd 8443:443
# 브라우저: https://localhost:8443 (인증서 경고는 "고급 → 계속"으로 통과)
```

### 5-4. 로그인 실패 → 비밀번호 직접 리셋

초기 비밀번호로 로그인이 계속 실패했습니다. (이중 설치 과정에서 비밀번호 해시와
초기 secret이 어긋났을 가능성) → `argocd-secret`의 해시를 직접 패치해서 해결.

```bash
# admin 비밀번호를 'password'로 리셋 (bcrypt 해시 패치, 재시작 불필요)
kubectl -n argocd patch secret argocd-secret -p \
  '{"stringData": {"admin.password": "$2a$10$rRyBsGSHK6.uc8fntPwVIuLVHgsAhAX7TcdrqW/RADU0uh7CaChLa", "admin.passwordMtime": "2026-06-10T05:16:00Z"}}'
```

> ⚠️ 로그인 후 **반드시** User Info → UPDATE PASSWORD 에서 비밀번호를 변경할 것!

### 5-5. Application 등록 — ComparisonError

`Infra/config/argocd/argocd-app.yaml`을 만들어 등록했습니다.

```bash
kubectl apply -f Infra/config/argocd/argocd-app.yaml
```

그런데 UI에서 에러 발생:

```text
ComparisonError: ... unable to resolve 'main' to a commit SHA
```

**원인**: `repoURL`(`https://github.com/sk-final-deepfake/infra.git`)에 해당하는
**저장소가 아직 GitHub에 존재하지 않음**. Argo CD는 Git 저장소를 읽어 클러스터와 비교하는
도구라서 저장소 푸시가 선행돼야 합니다.
