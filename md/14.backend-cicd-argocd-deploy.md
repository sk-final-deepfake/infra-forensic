# 13. Backend 재배포 + CI/CD — GitHub Actions + Argo CD GitOps (따라하기)

> **대상 코드:** `backend-forensic` (Spring Boot 3.5, Java 17)  
> **Infra 매니페스트:** `infra-forensic/config/k8s/backend/` → Argo CD가 watch  
> **전제:** EKS · ECR · Argo CD(Auto Sync) · ai-fastapi CI/CD 완료 ([12번 문서](./ai-cicd-argoCD-deploy.md) 기준)
>
> **명령어 실행 환경**
>
> - **Windows PC (PowerShell):** 본 문서의 `powershell` 블록 사용
> - PowerShell 여러 줄 이어쓰기: 줄 끝에 **백틱** ``` 사용

이 문서는 backend를 **수동 1회 재배포**한 뒤, **GitHub Actions → ECR → infra-forensic → Argo CD** 자동 배포(GitOps)를 연결합니다.  
ai-fastapi(12번 문서)와 **완전히 같은 패턴**입니다. 대상만 backend로 바뀝니다.

## 0. 한눈에 보는 배포 흐름

```text
backend-forensic main push
  → GitHub Actions (gradle 빌드는 Docker 안에서)
  → ECR push  forenshield-backend:<git-sha> + :latest
  → infra-forensic  config/k8s/backend/deployment.yaml  image tag 갱신
  → Argo CD Auto Sync
  → EKS backend Pod Rolling Update (replicas 2, 무중단)
```

### ai-fastapi와의 차이


| 항목        | ai-fastapi (12번)                         | backend (본 문서)                                      |
| --------- | ---------------------------------------- | --------------------------------------------------- |
| ECR 레포    | `forenshield-ai-fastapi`                 | `forenshield-backend` (이미 존재 ✅)                     |
| 매니페스트     | `config/k8s/ai-fastapi/`                 | `config/k8s/backend/` (이미 존재 ✅)                     |
| OIDC Role | `forenshield-ai-forensic-github-actions` | `**forenshield-backend-github-actions` 신규 생성 필요 ❌** |
| 빌드        | pip + Dockerfile                         | gradle 멀티스테이지 Dockerfile (이미 존재 ✅)                  |
| replicas  | 1                                        | 2 (Rolling Update 무중단)                              |
| 헬스        | `/health`                                | `/actuator/health`                                  |


> Argo CD `automated` sync는 12번 문서에서 이미 켰으므로 **별도 설정 불필요**합니다.

## 2. 수동 1회 재배포

CI/CD 전에 현재 코드로 한 번 수동 배포해서 빌드·기동이 정상인지 확인합니다.

> Docker Desktop이 켜져 있어야 합니다 (트레이 아이콘 초록색).  
> PowerShell **새 창**이면 환경 변수부터 다시 설정하세요.

```powershell
$env:AWS_PROFILE = "forenshield"
$env:AWS_REGION = "ap-northeast-2"
$env:ECR_REGISTRY = "877044078824.dkr.ecr.ap-northeast-2.amazonaws.com"
$env:IMAGE_TAG = "manual-redeploy-1"

cd C:\FINAL\backend-forensic

# ECR 로그인
aws ecr get-login-password --region $env:AWS_REGION | `
  docker login --username AWS --password-stdin $env:ECR_REGISTRY

# 빌드 (gradle 빌드가 Docker 안에서 돌아 5~10분 걸릴 수 있음)
docker build -t "forenshield-backend:$env:IMAGE_TAG" .

# 태그 + push
docker tag "forenshield-backend:$env:IMAGE_TAG" `
  "$env:ECR_REGISTRY/forenshield-backend:$env:IMAGE_TAG"
docker tag "forenshield-backend:$env:IMAGE_TAG" `
  "$env:ECR_REGISTRY/forenshield-backend:latest"
docker push "$env:ECR_REGISTRY/forenshield-backend:$env:IMAGE_TAG"
docker push "$env:ECR_REGISTRY/forenshield-backend:latest"
```

배포 (deployment.yaml이 `:latest` + `imagePullPolicy: Always`이므로 restart만 하면 됨):

```powershell
kubectl rollout restart deployment/backend -n forenshield
kubectl rollout status deployment/backend -n forenshield --timeout=300s
kubectl get pods -n forenshield -l app=backend
```

헬스 확인:

```powershell
kubectl run curl-be --restart=Never -n forenshield --image=curlimages/curl -- curl -s http://backend:8080/actuator/health
Start-Sleep -Seconds 8
kubectl logs curl-be -n forenshield
kubectl delete pod curl-be -n forenshield --wait=false
```

기대: `{"status":"UP"}` 비슷한 응답. 여기까지 OK면 CI/CD 연동으로 진행합니다.

## 3. GitHub OIDC IAM Role 생성 (1회)

ai-forensic용 Role(`forenshield-ai-forensic-github-actions`)은 Trust가 **ai-forensic repo 전용**이라 backend에서 못 씁니다.  
backend용 Role을 새로 만듭니다. (GitHub OIDC Provider는 계정에 이미 등록돼 있음 ✅)

> **방법 A(AWS 콘솔)** 또는 **방법 B(PowerShell CLI)** 중 하나만 하면 됩니다. 결과물(Role)은 완전히 동일합니다.  
> 콘솔은 GitHub OIDC용 입력 폼이 있어 JSON을 직접 쓸 필요가 없습니다.

### 방법 A — AWS 콘솔 (쉬움)

**A.1 Role 생성 시작**

1. AWS Console → **IAM** → 왼쪽 메뉴 **Roles(역할)** → **Create role(역할 생성)**
2. Trusted entity type에서 **Web identity(웹 자격 증명)** 선택

**A.2 OIDC Provider 폼 입력**


| 항목                  | 선택/입력 값                                             |
| ------------------- | --------------------------------------------------- |
| Identity provider   | `token.actions.githubusercontent.com` (드롭다운에 이미 있음) |
| Audience            | `sts.amazonaws.com`                                 |
| GitHub organization | `sk-final-deepfake`                                 |
| GitHub repository   | `backend-forensic`                                  |
| GitHub branch       | **비워두기** (모든 브랜치 허용 = `:`*와 동일)                     |


이 폼이 Trust Policy JSON을 자동 생성합니다. → **Next**

**A.3 권한 정책 연결**

1. 검색창에 `AmazonEC2ContainerRegistryPowerUser` 입력
2. 체크박스 선택 → **Next**

**A.4 이름 짓고 생성**


| 항목          | 값                                                        |
| ----------- | -------------------------------------------------------- |
| Role name   | `forenshield-backend-github-actions`                     |
| Description | (선택) GitHub Actions OIDC role for backend-forensic CI/CD |


생성 전 Trust policy 미리보기에 방법 B의 JSON과 같은 내용이 자동으로 들어가 있습니다. → **Create role**

**A.5 ARN 복사**

생성된 Role 클릭 → 상단 **ARN** 복사:

```text
arn:aws:iam::877044078824:role/forenshield-backend-github-actions

arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPowerUser (이건 내가 복붙한거)
```

이 값을 4장에서 GitHub Secret `AWS_ROLE_ARN`에 넣습니다. **방법 A로 했으면 §3.1~3.3은 건너뛰고 4장으로 진행하세요.**

### 방법 B — PowerShell CLI

#### 3.1 Trust Policy 파일 생성

```powershell
@'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::877044078824:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:sk-final-deepfake/backend-forensic:*"
        }
      }
    }
  ]
}
'@ | Set-Content -Path "$env:TEMP\backend-gha-trust.json" -Encoding utf8
```

#### 3.2 Role 생성 + ECR 정책 연결

```powershell
aws iam create-role `
  --role-name forenshield-backend-github-actions `
  --assume-role-policy-document "file://$env:TEMP\backend-gha-trust.json" `
  --description "GitHub Actions OIDC role for backend-forensic CI/CD"

aws iam attach-role-policy `
  --role-name forenshield-backend-github-actions `
  --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPowerUser
```

#### 3.3 확인

```powershell
aws iam get-role --role-name forenshield-backend-github-actions `
  --query "Role.Arn" --output text
```

출력된 ARN을 기록합니다:

```text
arn:aws:iam::877044078824:role/forenshield-backend-github-actions
```

> EKS 권한은 **불필요**합니다. 배포는 Argo CD가 담당하므로 CI에는 ECR push 권한만 있으면 됩니다.  
> (12번 문서에서 `Wait for rollout` 단계를 제거한 것과 같은 이유)

## 4. GitHub Secrets 등록

`backend-forensic` 저장소 → **Settings → Secrets and variables → Actions → New repository secret**


| Secret             | 값                                                                    |
| ------------------ | -------------------------------------------------------------------- |
| `AWS_ROLE_ARN`     | `arn:aws:iam::877044078824:role/forenshield-backend-github-actions`  |
| `INFRA_REPO_TOKEN` | `infra-forensic` write 권한 PAT — **ai-forensic에 등록한 것과 같은 토큰 재사용 가능** |


> PAT 새로 만들 필요 없음: ai-forensic CI/CD 때 만든 PAT가 `infra-forensic` write 권한이면 그대로 복사해서 등록하면 됩니다.  
> `AWS_REGION`, `INFRA_REPO`는 워크플로 env에 하드코딩하므로 Secret 불필요.

## 5. 워크플로 파일 추가

`backend-forensic/.github/workflows/deploy.yml`

```yaml
name: Backend Deploy (ECR + Argo CD GitOps)

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  id-token: write
  contents: read

env:
  AWS_REGION: ap-northeast-2
  ECR_REPOSITORY: forenshield-backend
  INFRA_REPO: sk-final-deepfake/infra-forensic
  DEPLOYMENT_PATH: config/k8s/backend/deployment.yaml

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout backend-forensic
        uses: actions/checkout@v4

      - name: Configure AWS credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to ECR
        id: ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build and push Docker image
        env:
          IMAGE_TAG: ${{ github.sha }}
          ECR_REGISTRY: ${{ steps.ecr.outputs.registry }}
        run: |
          docker build \
            -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG \
            -t $ECR_REGISTRY/$ECR_REPOSITORY:latest \
            .
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest

      - name: Checkout infra-forensic
        uses: actions/checkout@v4
        with:
          repository: ${{ env.INFRA_REPO }}
          token: ${{ secrets.INFRA_REPO_TOKEN }}
          path: infra

      - name: Update image tag in deployment manifest
        env:
          IMAGE_TAG: ${{ github.sha }}
          ECR_REGISTRY: ${{ steps.ecr.outputs.registry }}
        run: |
          NEW_IMAGE="$ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG"
          FILE="infra/$DEPLOYMENT_PATH"
          sed -i "s|image: .*forenshield-backend:.*|image: $NEW_IMAGE|" "$FILE"
          grep "image:" "$FILE"

      - name: Push to infra-forensic (triggers Argo CD)
        working-directory: infra
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add $DEPLOYMENT_PATH
          git diff --staged --quiet && echo "No manifest change" && exit 0
          git commit -m "ci: backend image → ${{ github.sha }}"
          git push origin master
```

> **주의:** gradle 테스트는 Dockerfile에서 `-x test`로 건너뜁니다.  
> CI에서 테스트를 돌리고 싶으면 `Build and push` 전에 `./gradlew test` step을 추가하세요 (JDK 17 setup 필요).

push:

```powershell
cd C:\FINAL\backend-forensic
git add .github/workflows/deploy.yml
git commit -m "ci: add GitHub Actions deploy workflow (ECR + Argo CD GitOps)"
git push origin main
```

> ⚠️ 이 push 자체가 **첫 CI/CD 실행을 트리거**합니다. Secrets를 먼저 등록하세요 (4장).

## 7. 트러블슈팅


| 증상                                 | 원인                               | 해결                                                                                  |
| ---------------------------------- | -------------------------------- | ----------------------------------------------------------------------------------- |
| CI `AssumeRoleWithWebIdentity` 실패  | Trust의 repo 경로 오타 · Role ARN 오입력 | 3.1 Trust `repo:sk-final-deepfake/backend-forensic:*` 확인, Secret `AWS_ROLE_ARN` 재확인 |
| CI ECR push 403                    | Role에 ECR 정책 미연결                 | 3.2 `AmazonEC2ContainerRegistryPowerUser` attach 확인                                 |
| CI infra push 403                  | PAT 권한 부족 · 만료                   | `repo` scope(또는 infra-forensic Contents R/W) PAT 재발급                                |
| `eks:DescribeCluster` AccessDenied | 워크플로에 kubectl step이 남아 있음        | GitOps에선 불필요 — 해당 step 제거 (본 문서 워크플로엔 없음)                                           |
| Argo CD `OutOfSync` 유지             | Auto Sync 꺼짐                     | `argocd-app.yaml` `syncPolicy.automated` 확인 또는 UI에서 수동 Sync                         |
| Pod `CrashLoopBackOff`             | prod 설정·Secret 누락                | `kubectl logs -n forenshield -l app=backend --tail=100`, Secret 5종 확인               |
| 빌드만 10분+                           | gradle 의존성 캐시 없음                 | 정상 (Docker 레이어 캐시가 없는 첫 빌드). 필요시 GHA cache 추가                                       |
| 새 이미지인데 동작 그대로                     | 구 Pod로 라우팅                       | `kubectl rollout status deployment/backend -n forenshield` 완료 대기                    |


로그 · 롤백:

```powershell
kubectl logs -n forenshield -l app=backend --tail=200 -f
kubectl describe pod -n forenshield -l app=backend

# 롤백 (직전 버전)
kubectl rollout undo deployment/backend -n forenshield
# 정석 롤백: infra-forensic에서 image tag를 이전 sha로 revert 후 push
```
