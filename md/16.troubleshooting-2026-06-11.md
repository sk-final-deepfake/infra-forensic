## 1. AWS CLI · IAM · OIDC

### 1-1. NAT Gateway IP 조회 — `InvalidParameter` / `<VPC_ID>`

**에러:**

```text
Value '[<VPC_ID>]' at 'filters.1.member.value' failed to satisfy constraint
```

**원인:** 문서의 `<VPC_ID>` 플레이스홀더를 **문자 그대로** 넣음. `export VPC_ID=...` 해도 명령에 `$VPC_ID`를 안 쓰면 소용없음.

**해결:**

```bash
export VPC_ID=$(aws eks describe-cluster --name forenshield --region ap-northeast-2 \
  --query 'cluster.resourcesVpcConfig.vpcId' --output text)

aws ec2 describe-nat-gateways \
  --filter "Name=vpc-id,Values=$VPC_ID" "Name=state,Values=available" \
  --query 'NatGateways[*].NatGatewayAddresses[*].PublicIp' --output text \
  --region ap-northeast-2
```

**우리 환경 결과:** `3.37.251.141` → GPU 방화벽 `:8000` Inbound 허용 대상.

---

### 1-2. `get-role` — `NoSuchEntity`

**에러:**

```text
The role with name forenshield-ai-forensic-github-actions cannot be found.
```

**원인:** `get-role`은 **이미 만든 Role 조회** 명령. `create-role`을 안 했음.

**해결:** Trust JSON + `aws iam create-role` → `attach-role-policy` (ECR PowerUser) 후 ARN을 GitHub Secret `AWS_ROLE_ARN`에 등록.

```text
arn:aws:iam::877044078824:role/forenshield-ai-forensic-github-actions
arn:aws:iam::877044078824:role/forenshield-frontend-github-actions
```

---

### 1-3. Windows — `file:///tmp/trust-github-frontend.json` 없음

**에러:**

```text
Unable to load paramfile file:///tmp/trust-github-frontend.json: [Errno 2] No such file or directory
```

**원인:** Linux 문서의 `/tmp/...` 경로. Windows Git Bash에는 파일이 없음. JSON 안의 `${AWS_ACCOUNT_ID}`도 **자동 치환 안 됨**.

**해결:**

```bash
# Infra 폴더 기준
aws iam update-assume-role-policy \
  --role-name forenshield-frontend-github-actions \
  --policy-document file:///c/Final_Project/Infra/tmp/trust-github-frontend.json
```

develop CI 추가 시 trust policy에 **main + develop** 브랜치 모두 허용.

---

### 1-4. GitHub OIDC vs `AWS_ROLE_ARN` (헷갈림)

| 항목 | ARN 예시 | 역할 |
|------|----------|------|
| OIDC Provider | `...oidc-provider/token.actions.githubusercontent.com` | AWS에 GitHub 신뢰 등록 (1회) |
| **Secret `AWS_ROLE_ARN`** | `...role/forenshield-frontend-github-actions` | CI가 AssumeRole 할 **Role** |

## 2. GPU 서버 (AI Gateway)

### 2-1. `source ~/venv/forenshield/bin/activate` 실패

**원인:** `python3 -m venv ~/venv/forenshield` 안 함.

**해결:**

```bash
python3 -m venv ~/venv/forenshield
source ~/venv/forenshield/bin/activate
cd /opt/forenshield/ai-gateway
pip install -r requirements.txt
```

---

### 2-2. `ModuleNotFoundError: No module named 'fastapi'`

**원인:**

1. `pip install -r requirements.txt` 안 함  
2. `/usr/bin/uvicorn`(시스템) 사용 → venv 패키지 미사용

**해결:**

```bash
source ~/venv/forenshield/bin/activate
pip install -r requirements.txt
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000
```

---

### 2-3. systemd `status=217/USER`

**에러:** `Process ... (code=exited, status=217/USER)`

**원인:** service 파일 `User=ubuntu`, `ExecStart=/home/ubuntu/...` 인데 실제 계정은 **`sk4team`**.

**해결:** `/etc/systemd/system/forenshield-ai-gateway.service`

```ini
User=sk4team
ExecStart=/home/sk4team/venv/forenshield/bin/python -m uvicorn app.main:app --host 0.0.0.0 --port 8000
```

```bash
sudo systemctl daemon-reload
sudo systemctl restart forenshield-ai-gateway
```

---

### 2-4. GPU에서 `git clone` — Username/Password

**원인:** `ai-forensic` private repo.

**해결:**

- Username: GitHub 아이디  
- Password: **PAT** (`ghp_...`), 웹 비밀번호 ❌  
- 또는 PC에서 `scp`로 코드 복사

---

### 2-5. GPU `aws: command not found`

**원인:** GPU 서버에 AWS CLI 미설치.

**해결 (선택):** Mock `/health`만이면 당장 불필요. S3 연동 시:

```bash
sudo snap install aws-cli --classic
# 또는 공식 AWS CLI v2 installer
```

## 3. Git · infra-forensic · frontend

### 3-1. `git pull` — 로컬 변경 / untracked 충돌

**에러:**

```text
error: Your local changes to ... app-config.yaml would be overwritten by merge
error: The following untracked working tree files would be overwritten by merge:
        config/k8s/ai-fastapi/deployment.yaml
```

**해결 (원격 기준 맞출 때):**

```bash
git restore config/k8s/app-config.yaml
rm -rf config/k8s/ai-fastapi
git pull origin master --no-edit
```

---

### 3-2. `git push rejected` + vim `MERGING`

**증상:** `pull` 후 `Waiting for your editor to close the file... Vim: Error` → `master|MERGING`

**원인:** merge 커밋 메시지용 vim이 Git Bash에서 동작 안 함.

**해결:**

```bash
git commit -m "merge: origin/master"
git push origin master
```

**예방:**

```bash
git pull origin master --no-edit
```

frontend `develop`도 동일:

```bash
git pull origin develop --no-edit
git commit -m "merge: origin/develop"
git push origin develop
```

## 4. Argo CD · K8s 매니페스트

### 4-1. `kubectl apply argocd-app.yaml` — YAML line 17

**에러:**

```text
yaml: line 17: did not find expected key
```

**원인:** `syncPolicy:` 들여쓰기가 3칸 (`destination`은 2칸).

**해결:**

```yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: forenshield
  syncPolicy:          # ← destination 과 같은 들여쓰기
    automated:
      prune: true
      selfHeal: true
```

## 5. GitHub Actions CI/CD

### 5-1. AI workflow — `eks:DescribeCluster` AccessDenied

**에러:**

```text
User: ...assumed-role/forenshield-ai-forensic-github-actions/GitHubActions
is not authorized to perform: eks:DescribeCluster
```

**원인:** Role에 ECR만 있고 EKS 권한 없음. `kubectl rollout status` 단계에서 실패.

**해결 (GitOps 방식):**

- workflow에서 **EKS rollout step 제거**  
- ECR push → infra-forensic push → **Argo CD Auto Sync**만 사용

infra push까지 성공했으면 배포는 이미 진행 중일 수 있음.

---

### 5-2. `INFRA_REPO_TOKEN`

- AWS CLI로 만들 수 **없음** → GitHub PAT (`repo` scope)  
- Classic: **read+write** = `repo` 체크  
- `ai-forensic` / `frontend-deepfake` 저장소 Secrets에 등록

## 7. EKS Pod · Rolling Update

### 7-1. Pod 두 개 — `0/2 Error` + `2/2 Running`

```text
frontend-5c9d87c7bd-...   0/2   Error
frontend-78b897b955-...   2/2   Running
```

**원인:** Rolling Update 중 **이전 배포 실패 Pod** 잔해 + **새 ReplicaSet 성공 Pod**.

**판단:**

- **새 Pod `2/2 Running`** → 배포 성공  
- Error Pod는 곧 사라짐  
- 이미지가 CI SHA(`e1cf679...`)로 바뀌었으면 GitOps 정상 동작

```bash
kubectl get pods -n forenshield -l app=frontend
kubectl get deployment frontend -n forenshield \
  -o jsonpath="{.spec.template.spec.containers[?(@.name=='nextjs')].image}{'\n'}"
```

## 8. GitOps 개념 헷갈림

### 8-1. Argo CD `Synced`인데 새 이미지가 안 보임

**오해:** `Synced` = ECR 최신 이미지 반영됨

**실제:**

```text
Synced = infra-forensic Git == EKS 클러스터
ECR에 push만 하고 Git의 deployment.yaml image tag를 안 바꾸면 Argo는 모름
```

**해결:**

1. CI가 infra-forensic `deployment.yaml` image tag 갱신 + push  
2. 또는 수동으로 tag 변경 (`manual-main-1`, `develop-<sha>` 등) 후 infra push  
3. Argo CD Auto Sync → Pod 교체

### 8-2. main vs develop 프론트 CI

| 브랜치 | workflow | ECR 태그 | 배포 대상 |
|--------|----------|----------|-----------|
| `main` | `deploy.yml` | `<sha>`, `latest` | 동일 production frontend |
| `develop` | `deploy-develop.yml` | `develop-<sha>` | **동일** production frontend |

develop만 **별도 URL**로 쓰려면 `frontend-develop` Deployment를 infra에 추가해야 함.

### 8-3. AI Gateway vs EKS ai-fastapi

| | GPU AI Gateway | EKS ai-fastapi |
|--|----------------|----------------|
| 위치 | GPU 서버 (welabs) | EKS |
| 배포 | clone + systemd (수동) | CI + Argo CD |
| 역할 | 실제 추론 (PyTorch) | RabbitMQ·오케스트레이션 |

Argo CD는 **EKS만** 관리 → GPU에는 코드를 **직접** clone/scp.
