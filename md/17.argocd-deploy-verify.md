## 1. Argo CD가 뭔지 — bomin 4~7과 관계

| 방식 | bomin 4~7 (지금까지) | Argo CD |
| --- | --- | --- |
| 배포 | `kubectl apply -f k8s/` 수동 | Git 저장소 YAML ↔ 클러스터 **자동 동기화** |
| 확인 | `kubectl get pods`, curl E2E | **웹 UI**에서 앱 트리·Healthy·Sync 상태 |
| 변경 | YAML 수정 → apply 다시 | Git push → Argo CD가 감지 → Sync |

**중요:** Argo CD는 **Application(앱)** 을 등록해야 UI에 forenshield 배포가 보입니다.  
`kubectl apply`만 하고 Application을 안 만들면 Argo CD Pod는 떠 있어도 **대시보드가 비어 있습니다.**

```text
GitHub (k8s/*.yaml)
       │
       ▼  Argo CD가 주기적으로 diff
  Application CR  ──►  EKS ns: forenshield
       │                    ├─ backend Deployment
       │                    ├─ frontend Deployment
       │                    └─ Ingress ...
       ▼
  Argo CD UI (Healthy / Synced / OutOfSync)
```

## 5. Application 등록 (UI에서 forenshield 보이게 하기)

수동으로 apply한 리소스를 Argo CD가 **관리·표시**하려면 Application을 하나 이상 만들어야 합니다.

### 5.1 등록 전 체크 — Git에 manifest push

```powershell
# 프론트 예시 — k8s 폴더가 원격에 있는지
cd C:\FINAL\frontend-deepfake
git status
git ls-remote origin HEAD
# k8s/deployment.yaml, service.yaml, nginx-config.yaml 이 원격 브랜치에 있어야 함
```

없으면 커밋·push 후 Application 생성:

```text
frontend-deepfake/k8s/
├── deployment.yaml
├── service.yaml
└── nginx-config.yaml
```

백엔드·Ingress도 동일 — **팀에서 manifest를 모아둔 Git path**를 정한 뒤 Application `path`에 넣습니다.

### 5.2 UI로 Application 만들기 (프론트 예시)

1. Argo CD UI → **+ NEW APP**
2. 아래 표대로 입력:

| 필드 | 값 |
| --- | --- |
| Application Name | `forenshield-frontend` |
| Project | `default` |
| Sync Policy | Manual (처음엔 Manual 권장 — 자동 Sync는 익숙해진 뒤) |
| Repository URL | `https://github.com/sk-final-deepfake/frontend-deepfake.git` |
| Revision | `main` (또는 k8s가 있는 브랜치) |
| Path | `k8s` |
| Cluster URL | `https://kubernetes.default.svc` (in-cluster) |
| Namespace | `forenshield` |

3. **CREATE** → **SYNC** 클릭

> 이미 `kubectl apply`로 같은 리소스가 있으면 **Sync 시 merge/OutOfSync** 가 날 수 있습니다.  
> 첫 Sync 전에 `kubectl get deploy frontend -n forenshield -o yaml` 과 Git YAML diff를 맞추거나, 팀 정책에 따라 **기존 리소스 삭제 후 Argo CD만 관리**할지 결정하세요.

### 5.3 Application YAML (CLI/GitOps용 — 복사해서 apply)

`argocd/applications/forenshield-frontend.yaml` 등으로 레포에 두고 apply해도 됩니다.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: forenshield-frontend
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/sk-final-deepfake/frontend-deepfake.git
    targetRevision: main        # k8s가 있는 브랜치로 변경
    path: k8s
  destination:
    server: https://kubernetes.default.svc
    namespace: forenshield
  syncPolicy:
    syncOptions:
      - CreateNamespace=false   # forenshield ns는 이미 존재
```

```powershell
kubectl apply -f forenshield-frontend-application.yaml
```

### 5.4 백엔드 · Ingress Application (팀 manifest path에 맞게)

| Application 이름 | repoURL (예) | path (팀 확인) |
| --- | --- | --- |
| `forenshield-backend` | `backend-forensic` 또는 infra 레po | `k8s/backend` 또는 `Infra/k8s/backend` |
| `forenshield-ingress` | infra 레po | `k8s/ingress.yaml` 있는 디렉터리 |

Ingress는 파일 1개만 있어도 `path`를 그 파일이 있는 폴더로 지정하면 됩니다.

### 5.5 Private GitHub 레po인 경우

Argo CD가 clone 못 하면 Application이 **Unknown / ComparisonError** 됩니다.

1. GitHub **Personal Access Token** (repo read)
2. Argo CD UI → **Settings → Repositories → Connect Repo**
3. 또는 Secret:

```powershell
kubectl create secret generic repo-frontend-deepfake -n argocd `
  --from-literal=type=git `
  --from-literal=url=https://github.com/sk-final-deepfake/frontend-deepfake.git `
  --from-literal=password=<GITHUB_PAT> `
  --from-literal=username=git
```

## 6. UI에서 확인하는 법

Application 카드/상세 화면에서 볼 것:

### 6.1 상태 뱃지

| 표시 | 의미 | forenshield에서 기대 |
| --- | --- | --- |
| **Health: Healthy** | Pod/Deployment 등 정상 | backend 2/2, frontend 2/2 컨테이너 Ready |
| **Sync: Synced** | Git manifest = 클러스터 | Git과 수동 apply 내용이 같을 때 |
| **Sync: OutOfSync** | Git ≠ 클러스터 | 이미지 태그·replicas 등 drift — Sync 또는 Git 수정 |
| **Progressing** | 롤아웃 중 | 배포 직후 잠깐 |
| **Degraded** | Pod 실패·Probe 실패 | `kubectl describe pod` 로 원인 |

### 6.2 Resource tree (트리 뷰)

클릭하면 계층이 보입니다:

```text
forenshield-frontend (Application)
└─ Deployment/frontend
   └─ ReplicaSet
      └─ Pod (nginx + nextjs)  ← 각각 녹색 체크
└─ Service/frontend
└─ ConfigMap/nginx-config      ← nginx-config를 Git/k8s에 포함한 경우
```

backend Application이면 `Deployment/backend`, `Service/backend` + envFrom Secret/ConfigMap(외부 관리) 참조.

### 6.3 Live Manifest vs Git diff

- **APP DIFF** — Git에 있는 YAML vs 클러스터 실제 상태  
- 수동 `kubectl edit` 했으면 OutOfSync + diff에 차이 표시

### 6.4 Events / Sync History

- **History and rollback** — 이전 Sync 버전으로 롤백 가능  
- 실패 시 **Sync failed** 메시지에 RBAC, repo 접근, YAML 문법 오류 등 표시

## 9. 자주 막히는 곳

| 증상 | 원인 | 해결 |
| --- | --- | --- |
| UI에 앱이 없음 | Application 미등록 | [5번](#5-application-등록-ui에서-forenshield-보이게-하기) |
| `ComparisonError` / repo access | Private repo, PAT 없음 | [5.5](#55-private-github-레po인-경우) |
| OutOfSync (image 태그) | Git은 `:latest` 고정, 수동으로 다른 태그 push | Git에 tag/commit SHA 반영 또는 Sync |
| Sync OK인데 502 | Pod는 Healthy지만 앱 설정 오류 | `kubectl logs -n forenshield -l app=backend` |
| frontend만 Degraded | nginx-config ConfigMap 없음 | `k8s/nginx-config.yaml` Git 포함 + Sync |
| RabbitMQ 안 보임 | Helm으로 설치, Application 없음 | 별도 Application(Helm chart) 또는 수동 리소스는 UI 미표시 |
| port-forward 끊김 | 터미널 종료 | `kubectl port-forward svc/argocd-server -n argocd 8443:443` 재실행 |
