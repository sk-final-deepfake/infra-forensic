## 2. K8s Secret · ConfigMap 적용

백엔드/프론트 Pod가 읽을 **접속 정보(비밀번호 포함)** 를 클러스터에 먼저 넣어둡니다.
이미 `Infra/config/`에 스크립트가 다 준비되어 있어서 명령 3개면 끝납니다.

```bash
cd Infra

# 1) 비밀번호 파일 생성 (Git 커밋 금지!)
cp config/secrets.env.example config/secrets.env
# 열어서 POSTGRES_PASSWORD, REDIS_PASSWORD, RABBITMQ_PASSWORD 실제 값 입력

# 2) 일괄 적용
bash config/apply-settings.sh
```

적용되면 아래 리소스가 생깁니다. **이 이름들을 그대로 Deployment에서 참조**하므로 외워둘 필요는 없고, 존재만 확인하면 됩니다.

```bash
kubectl get secret,configmap,sa -n forenshield
```


| 이름                     | 종류             | 들어있는 것                                                        |
| ---------------------- | -------------- | ------------------------------------------------------------- |
| `db-credentials`       | Secret         | `POSTGRES_HOST/USER/PASSWORD/DB`                              |
| `redis-credentials`    | Secret         | `REDIS_HOST/PASSWORD`                                         |
| `rabbitmq-credentials` | Secret         | `RABBITMQ_USER/PASSWORD`                                      |
| `s3-config`            | Secret         | `S3_EVIDENCE_BUCKET`, `AWS_REGION`                            |
| `app-config`           | ConfigMap      | `SPRING_PROFILES_ACTIVE=prod`, `RABBITMQ_HOST`, `REDIS_SSL` 등 |
| `frontend-config`      | ConfigMap      | `NEXT_PUBLIC_API_URL`, `NODE_ENV`                             |
| `forenshield-app`      | ServiceAccount | S3 접근용 IAM Role 연결 (IRSA)                                     |


> **핵심 개념:** 코드에는 비밀번호·엔드포인트를 절대 하드코딩하지 않습니다. 코드는 `${POSTGRES_HOST}` 같은 **환경변수 이름만 알고**, 실제 값은 K8s가 Pod 실행 시 주입합니다. 그래서 로컬/운영 코드가 동일해집니다.

## 3. 백엔드 연결 (코드 수정)

`backend/backend-forensic`에서 작업합니다. 수정 포인트는 3개입니다.

### 3.1 build.gradle — 의존성 추가

PostgreSQL 드라이버는 이미 있고, **Redis · RabbitMQ · S3** 의존성을 추가합니다.

```groovy
dependencies {
    // ... 기존 의존성 유지 ...

    // 인프라 연동 추가
    implementation 'org.springframework.boot:spring-boot-starter-data-redis'
    implementation 'org.springframework.boot:spring-boot-starter-amqp'
    implementation platform('software.amazon.awssdk:bom:2.31.20')
    implementation 'software.amazon.awssdk:s3'
}
```

### 3.2 application-prod.yaml — 운영 프로파일 신규 작성

기존 `application.yaml`(H2)은 **로컬 개발용으로 그대로 두고**,
`src/main/resources/application-prod.yaml`을 새로 만듭니다.

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${POSTGRES_HOST}:5432/${POSTGRES_DB:forenshield}
    driver-class-name: org.postgresql.Driver
    username: ${POSTGRES_USER}
    password: ${POSTGRES_PASSWORD}
  jpa:
    hibernate:
      ddl-auto: validate        # 운영에서 update 절대 금지
    show-sql: false
  data:
    redis:
      host: ${REDIS_HOST}
      port: ${REDIS_PORT:6379}
      password: ${REDIS_PASSWORD}
      ssl:
        enabled: ${REDIS_SSL:true}   # ElastiCache는 TLS 필수
  rabbitmq:
    host: ${RABBITMQ_HOST}
    port: ${RABBITMQ_PORT:5672}
    username: ${RABBITMQ_USER}
    password: ${RABBITMQ_PASSWORD}
  servlet:
    multipart:
      max-file-size: 2GB
      max-request-size: 2GB
  autoconfigure:
    exclude: org.springframework.ai.vectorstore.pgvector.autoconfigure.PgVectorStoreAutoConfiguration

aws:
  region: ${AWS_REGION:ap-northeast-2}
  s3:
    evidence-bucket: ${S3_EVIDENCE_BUCKET}

management:
  endpoints:
    web:
      exposure:
        include: health,info
```

여기 등장하는 `${...}` 변수들은 전부 [2번](#2-k8s-secret--configmap-적용)에서 만든 Secret/ConfigMap이 주입합니다. **코드에 실제 값을 쓸 곳이 한 군데도 없는 게 정상**입니다.

> **ffmpeg 주의:** 현재 `application.yaml`의 `ffmpeg.path`가 Windows 경로(`C:\ffmpeg\...`)로 하드코딩되어 있습니다. prod에서는 컨테이너에 ffmpeg를 설치하고 경로를 `/usr/bin`으로 바꿔야 합니다 (아래 Dockerfile에서 처리).

> **첫 배포 시 스키마:** `ddl-auto: validate`라서 테이블이 없으면 기동에 실패합니다. 첫 1회는 로컬에서 RDS에 직접 스키마를 만들거나(VPN 경유 psql), 임시로 `ddl-auto: create` 1회 기동 후 `validate`로 되돌리세요. 장기적으로는 Flyway 도입 권장.

### 3.3 FileService — 로컬 저장 → S3 저장 전환

현재 `FileService`는 `uploads/original` 로컬 폴더에 저장합니다. Pod는 재시작하면 파일이 사라지므로 S3로 바꿔야 합니다. S3 클라이언트 설정 클래스를 추가하고:

```java
@Configuration
public class S3Config {

    @Value("${aws.region}")
    private String region;

    @Bean
    public S3Client s3Client() {
        // 자격증명 코드 없음 — IRSA(ServiceAccount)가 자동 주입
        return S3Client.builder()
                .region(Region.of(region))
                .build();
    }
}
```

`FileService` 저장 부분을 S3 업로드로 교체합니다.

```java
// 기존: Files.copy(file.getInputStream(), targetPath)
// 변경:
s3Client.putObject(
        PutObjectRequest.builder()
                .bucket(evidenceBucket)           // ${S3_EVIDENCE_BUCKET}
                .key("original/" + storedFileName)
                .contentType(file.getContentType())
                .build(),
        RequestBody.fromInputStream(file.getInputStream(), file.getSize()));
```

> **Access Key를 발급받지 마세요.** Pod가 `forenshield-app` ServiceAccount로 뜨면 AWS SDK가 IAM Role 자격증명을 알아서 가져갑니다. 이게 IRSA입니다. (로컬 테스트 시에는 `AWS_PROFILE=forenshield`로 동일 코드가 동작합니다.)

### 3.4 CORS 설정 (선택이지만 권장)

같은 도메인(`/api`) 아래로 서빙하므로 운영에서는 CORS 문제가 없지만, **로컬 개발**(프론트 `localhost:3000` → 백엔드 `localhost:8080`)을 위해 추가해 둡니다.

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("http://localhost:3000", "https://forensheildjangdochi.com")
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS");
    }
}
```

### 3.5 Dockerfile — 백엔드 레포 루트에 생성

`backend/backend-forensic/Dockerfile`:

```dockerfile
# ---- Stage 1: 빌드 ----
FROM eclipse-temurin:17-jdk AS builder
WORKDIR /app
COPY gradlew .
COPY gradle gradle
COPY build.gradle settings.gradle ./
RUN chmod +x gradlew && ./gradlew dependencies --no-daemon || true
COPY src src
RUN ./gradlew bootJar --no-daemon -x test

# ---- Stage 2: 실행 ----
FROM eclipse-temurin:17-jre
WORKDIR /app
# 메타데이터 추출용 ffmpeg (application-prod에서 ffmpeg.path=/usr/bin)
RUN apt-get update && apt-get install -y --no-install-recommends ffmpeg \
    && rm -rf /var/lib/apt/lists/*
RUN groupadd -r spring && useradd -r -g spring spring
USER spring:spring
COPY --from=builder /app/build/libs/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", \
  "-XX:+UseContainerSupport", \
  "-XX:MaxRAMPercentage=75.0", \
  "-jar", "app.jar"]
```

`application-prod.yaml`에 ffmpeg 경로 추가:

```yaml
ffmpeg:
  path: /usr/bin
```

## 4. 백엔드 빌드 → ECR → EKS 배포

### 4.1 로컬 스모크 테스트 (선택, 권장)

```bash
cd backend/backend-forensic
docker build -t forenshield-backend:local .

# 인메모리 H2 + /tmp 업로드 폴더로 기동
# (컨테이너는 비root 사용자라 /app에 쓰기 불가 → H2 파일 DB·uploads 폴더 생성 모두 실패함)
# Git Bash에서는 /tmp가 Windows 경로로 자동 변환되므로 MSYS_NO_PATHCONV=1 필수!
MSYS_NO_PATHCONV=1 docker run -p 8080:8080 \
  -e SPRING_DATASOURCE_URL="jdbc:h2:mem:demo;DB_CLOSE_DELAY=-1" \
  -e FILE_UPLOAD_DIR=/tmp/uploads \
  forenshield-backend:local

curl http://localhost:8080/actuator/health           # {"status":"UP"} 확인
```

> 환경변수 없이 그냥 실행하면 `Unable to determine Dialect without JDBC metadata`(H2 파일 DB 생성 실패) 또는 `Could not initialize folder for upload!`(uploads 폴더 생성 실패) 에러로 죽습니다. 운영(EKS)에서는 RDS 환경변수가 주입되고 FileService가 S3로 전환(3.3)되므로 발생하지 않는 문제입니다.

### 4.2 ECR Push

```bash
export AWS_REGION=ap-northeast-2
export ECR_REGISTRY=877044078824.dkr.ecr.ap-northeast-2.amazonaws.com
export IMAGE_TAG=$(git rev-parse --short HEAD)

# ECR 로그인 (12시간 유효)
aws ecr get-login-password --region $AWS_REGION \
  | docker login --username AWS --password-stdin $ECR_REGISTRY

# 빌드 · 태그 · Push
docker build -t $ECR_REGISTRY/forenshield-backend:$IMAGE_TAG \
             -t $ECR_REGISTRY/forenshield-backend:latest .
docker push $ECR_REGISTRY/forenshield-backend:$IMAGE_TAG
docker push $ECR_REGISTRY/forenshield-backend:latest
```

> PowerShell이면 `export` 대신 `$env:AWS_REGION="ap-northeast-2"` 형식으로, `$(git ...)`은 `$IMAGE_TAG = git rev-parse --short HEAD`로 바꾸면 됩니다.

### 4.3 K8s 매니페스트 작성

`Infra/k8s/backend/` 폴더를 만들고 아래 2개 파일을 작성합니다.

`Infra/k8s/backend/deployment.yaml`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
  namespace: forenshield
  labels:
    app: backend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: backend
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: backend
    spec:
      serviceAccountName: forenshield-app   # S3 IRSA — 2번에서 이미 생성됨
      nodeSelector:
        nodegroup: backend-ng
      containers:
        - name: backend
          image: 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com/forenshield-backend:latest
          imagePullPolicy: Always
          ports:
            - containerPort: 8080
          envFrom:
            - configMapRef:
                name: app-config           # SPRING_PROFILES_ACTIVE=prod 포함
            - secretRef:
                name: db-credentials
            - secretRef:
                name: redis-credentials
            - secretRef:
                name: rabbitmq-credentials
            - secretRef:
                name: s3-config
          readinessProbe:
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 60
            periodSeconds: 15
          resources:
            requests:
              cpu: 250m
              memory: 512Mi
            limits:
              cpu: 1000m
              memory: 1Gi
```

`Infra/k8s/backend/service.yaml`:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: backend
  namespace: forenshield
spec:
  type: ClusterIP
  selector:
    app: backend
  ports:
    - port: 8080
      targetPort: 8080
```

### 5.3 lib/api.ts — 공용 API 클라이언트 생성

`frontend/frontend-deepfake/lib/api.ts`:

```typescript
const API_BASE = process.env.NEXT_PUBLIC_API_URL ?? "http://localhost:8080"

export interface FileUploadResponse {
  success: boolean
  // 백엔드 FileUploadResponse DTO 필드에 맞게 보강
  [key: string]: unknown
}

export interface ApiError {
  success: false
  errorCode: string
  message: string
}

export async function uploadEvidence(file: File): Promise<FileUploadResponse> {
  const formData = new FormData()
  formData.append("file", file)

  const res = await fetch(`${API_BASE}/api/evidences/upload`, {
    method: "POST",
    body: formData, // Content-Type은 브라우저가 boundary 포함해 자동 설정
  })

  const body = await res.json()
  if (!res.ok) {
    throw new Error((body as ApiError).message ?? "파일 업로드에 실패했습니다.")
  }
  return body as FileUploadResponse
}
```

> 운영에서는 `NEXT_PUBLIC_API_URL=https://forensheildjangdochi.com/api`이므로 최종 호출 경로가 `/api/api/evidences/upload`로 중복되지 않도록 주의하세요. 백엔드 컨트롤러가 이미 `/api/evidences`를 쓰고 있으므로, **운영 빌드 시에는 `NEXT_PUBLIC_API_URL=https://forensheildjangdochi.com`(뒤에 `/api` 없이)으로 주입**하는 것이 정합합니다. 이 문서는 이 방식으로 통일합니다.

```yaml
            pathType: Prefix
```

**해결** (로컬 테스트용, EKS는 PostgreSQL):

```bash
docker run ... \

  -e MANAGEMENT_HEALTH_RABBIT_ENABLED=false
```

### 9.4 Ingress / ALB — 도메인 503

### 3-1. Ingress ADDRESS 비어 있음 + 503


### 9.9 백엔드 readiness probe Unhealthy (Argo CD Events)

### 증상

```text
Readiness probe failed: dial tcp ...:8080: connect: connection refused
Readiness probe failed: context deadline exceeded (Client.Timeout exceeded)
```

Argo CD Pod Events에 빨간 `Unhealthy` 다수 표시. 프론트 dev 배포와 **무관** — 백엔드 재시작 시 probe 타이밍 문제.

### 원인

Spring Boot + Hibernate(`ddl update`) + Redis/RabbitMQ 연결까지 **40~60초** 소요.
기존 설정: `initialDelaySeconds: 30`, `timeoutSeconds: 1`(기본값).

### 해결 (`backend/deployment.yaml`)

```yaml
startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 18    # 최대 ~3분 기동 허용
readinessProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3
livenessProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  periodSeconds: 15
  timeoutSeconds: 5
  failureThreshold: 3
```

```bash
kubectl apply -f Infra/config/k8s/backend/deployment.yaml
kubectl rollout restart deployment/backend -n forenshield
```

→ 기동 초반 `Startup probe failed` 2~3회는 정상. 이후 `1/1 Running`, health `UP`.

### 10.2 IAM 정책 수정 (ec2:DescribeRouteTables 추가)

### Role에 붙은 정책 확인

```bash
aws iam list-attached-role-policies \
  --role-name eksctl-forenshield-addon-iamserviceaccount-ku-Role1-dIwg9NBRLOZU
```

- **무슨 명령어?** IAM Role에 연결(attach)된 정책 목록을 조회합니다.
- **뭘 한 거?** ALB Controller가 쓰는 Role에 `AWSLoadBalancerControllerIAMPolicy` 하나만 붙어 있는 것을 확인했습니다.

### 정책 내용 확인

```bash
VER=$(aws iam get-policy --policy-arn <정책ARN> --query "Policy.DefaultVersionId" --output text)
aws iam get-policy-version --policy-arn <정책ARN> --version-id $VER \
  --query "PolicyVersion.Document" --output json > policy-doc.json
```

- **무슨 명령어?** `get-policy`는 정책의 현재 버전 ID를, `get-policy-version`은 해당 버전의 **실제 권한 목록(JSON)**을 보여줍니다.
- **뭘 한 거?** 정책 문서를 파일로 받아서 `ec2:DescribeRouteTables`가 **없는 것**을 확인하고, JSON에 그 권한을 추가했습니다.

### 새 정책 버전 생성

```bash
aws iam create-policy-version --policy-arn <정책ARN> \
  --policy-document file://policy-doc.json --set-as-default
```

- **무슨 명령어?** IAM 정책의 새 버전을 만들고 기본(default) 버전으로 지정합니다. Role에 다시 attach할 필요 없이 즉시 적용됩니다.
- **뭘 한 거?** `ec2:DescribeRouteTables`가 추가된 v2 버전을 만들어 권한 문제를 해결했습니다.

### 컨트롤러 재시작

```bash
kubectl rollout restart deployment/aws-load-balancer-controller -n kube-system
kubectl rollout status deployment/aws-load-balancer-controller -n kube-system
```

- **무슨 명령어?** `rollout restart`는 Deployment의 Pod를 순차 재생성하고, `rollout status`는 재시작이 끝날 때까지 진행 상황을 보여줍니다.
- **뭘 한 거?** 컨트롤러가 백오프(재시도 대기) 상태였기 때문에 재시작으로 즉시 재시도하게 했고, 그 결과 ALB가 생성됐습니다.

```bash
# 파일명에 공백·괄호가 있으면 따옴표로 감싸고 공백까지 정확히!
```

```bash
kubectl logs -n forenshield deployment/backend --since=10m
```

```bash
kubectl exec -n forenshield deployment/backend -- printenv | grep AWS_
```

- **무슨 명령어?** `jsonpath`로 Pod의 ServiceAccount 이름을 뽑고, `exec ... printenv`로 컨테이너 안의 환경변수를 확인합니다.
- **뭘 한 거?** `AWS_ROLE_ARN`, `AWS_WEB_IDENTITY_TOKEN_FILE`이 정상 주입돼 있음을 확인 → IRSA 설정은 문제없고, 순수하게 SDK 의존성 문제로 좁혔습니다.

### 백엔드 재빌드·재배포

```bash
cd /c/Final_Project/backend/backend-forensic

# build.gradle에 implementation 'software.amazon.awssdk:sts' 추가 후
docker build -t forenshield-backend .

aws ecr get-login-password --region ap-northeast-2 | \
  docker login --username AWS --password-stdin 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com

docker tag forenshield-backend:latest 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com/forenshield-backend:latest

```

- **무슨 명령어?**
  - `docker build`: Dockerfile로 이미지 빌드

  - `docker tag` / `docker push`: ECR 주소로 태그를 붙여 업로드
  - `kubectl rollout restart`: 새 이미지(`imagePullPolicy: Always`)로 Pod 재생성
- **뭘 한 거?** `sts` 의존성이 포함된 새 이미지를 배포 → S3 업로드가 성공했습니다.

### 10.6 Argo CD 설치·설정

### 설치

```bash
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

- **무슨 명령어?** Argo CD 공식 설치 매니페스트를 `argocd` 네임스페이스에 적용합니다.
- **뭘 한 거?** 설치 도중 `applicationsets.argoproj.io` CRD가 **256KB 어노테이션 제한** 때문에 실패했습니다.

### server-side apply로 재설치

```bash
kubectl apply --server-side --force-conflicts -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

- **무슨 명령어?**
  - `--server-side`: 적용 로직을 API 서버가 처리 → `last-applied-configuration` 어노테이션을 안 만들어서 큰 CRD도 적용 가능
  - `--force-conflicts`: 첫 설치(client-side)와의 필드 소유권 충돌을 강제로 덮어씀
- **뭘 한 거?** 실패했던 CRD까지 전부 적용 완료, 7개 Pod 모두 Running 확인.

### Pod 상태 확인

```bash
kubectl get pods -n argocd
```

- **무슨 명령어?** `argocd` 네임스페이스의 Pod 목록·상태를 확인합니다.
- **뭘 한 거?** server, repo-server, application-controller 등 7개가 전부 `Running`인 것을 확인했습니다.

### 초기 비밀번호 확인

```bash
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d; echo
```

- **무슨 명령어?** Secret의 값은 base64로 인코딩돼 있어서 `base64 -d`로 디코딩해야 실제 비밀번호가 나옵니다.
- **뭘 한 거?** 초기 admin 비밀번호를 확인했지만 로그인이 계속 실패했습니다.

### admin 비밀번호 강제 리셋

```bash
kubectl -n argocd patch secret argocd-secret -p \
  '{"stringData": {"admin.password": "$2a$10$rRyBsGSHK6.uc8fntPwVIuLVHgsAhAX7TcdrqW/RADU0uh7CaChLa", "admin.passwordMtime": "2026-06-10T05:16:00Z"}}'
```

- **무슨 명령어?** `kubectl patch`는 리소스의 일부 필드만 수정합니다. Argo CD는 `argocd-secret`의 `admin.password`(bcrypt 해시)를 로그인 검증에 사용합니다. 위 해시는 `password`라는 문자열의 bcrypt 값입니다.
- **뭘 한 거?** 비밀번호를 `password`로 리셋해서 로그인 문제를 해결했습니다. **로그인 후 UI에서 즉시 변경 필요!**

### UI 접속 (포트포워딩)

```bash
kubectl port-forward svc/argocd-server -n argocd 8443:443
# 브라우저: https://localhost:8443  (admin / 리셋한 비밀번호)
```

- **무슨 명령어?** 클러스터 내부 Service를 내 PC의 포트로 터널링합니다. 외부 노출 없이 안전하게 UI에 접속할 수 있습니다.
- **뭘 한 거?** Argo CD UI에 로그인했습니다.

### Application 등록

```bash

```

- **무슨 명령어?** Argo CD의 `Application` 커스텀 리소스를 등록합니다. "이 Git 저장소의 이 경로를 이 클러스터/네임스페이스와 동기화해라"라는 선언입니다.
- **뭘 한 거?** `forenshield` 앱을 등록했지만, `repoURL`의 저장소가 GitHub에 아직 없어서 `ComparisonError`가 떴습니다.
