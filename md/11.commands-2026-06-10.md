# 11. 오늘 사용한 명령어 정리 (2026-06-10)

> 오늘 배포 트러블슈팅에 사용한 **모든 명령어**를 순서대로 정리하고, 각 명령어가 무엇이고 무엇을 했는지 설명하는 문서입니다.
> 문제 해결 흐름(원인 분석)은 [10.troubleshooting-2026-06-10.md](./10.troubleshooting-2026-06-10.md)를 함께 보세요.

## 1. Ingress / ALB 상태 진단

### Ingress 목록 확인

```bash
kubectl get ingress -n forenshield
```

- **무슨 명령어?** `forenshield` 네임스페이스의 Ingress 리소스 목록을 보여줍니다.
- **뭘 한 거?** `ADDRESS` 칸이 비어 있는 것을 확인 → ALB가 아직 안 만들어졌다는 신호를 잡았습니다.

### Ingress 상세 + 이벤트 확인

```bash
kubectl describe ingress forenshield-ingress -n forenshield
```

- **무슨 명령어?** Ingress 하나의 상세 정보와 **이벤트(에러 로그)**를 보여줍니다.
- **뭘 한 거?** `FailedBuildModel ... not authorized to perform: ec2:DescribeRouteTables` 에러를 발견 → ALB Controller의 IAM 권한 부족이 원인임을 확인했습니다.

### ALB Controller 설치 여부 확인

```bash
kubectl get deployment -n kube-system aws-load-balancer-controller
```

- **무슨 명령어?** `kube-system` 네임스페이스에 ALB Controller가 떠 있는지 확인합니다.
- **뭘 한 거?** `2/2 Running`으로 컨트롤러 자체는 정상임을 확인 → 문제는 권한 쪽으로 좁혔습니다.

## 2. IAM 정책 수정 (ec2:DescribeRouteTables 추가)

### Role에 붙은 정책 확인

```bash
aws iam list-attached-role-policies \
  --role-name eksctl-forenshield-addon-iamserviceaccount-ku-Role1-dIwg9NBRLOZU
```

- **무슨 명령어?** IAM Role에 연결(attach)된 정책 목록을 조회합니다.
- **뭘 한 거?** ALB Controller가 쓰는 Role에 `AWSLoadBalancerControllerIAMPolicy` 하나만 붙어 있는 것을 확인했습니다.

### 정책 내용 확인

```bash
VER=$(aws iam get-policy --policy-arn <정책ARN> --query "Policy.DefaultVersionId" --output text)
aws iam get-policy-version --policy-arn <정책ARN> --version-id $VER \
  --query "PolicyVersion.Document" --output json > policy-doc.json
```

- **무슨 명령어?** `get-policy`는 정책의 현재 버전 ID를, `get-policy-version`은 해당 버전의 **실제 권한 목록(JSON)**을 보여줍니다.
- **뭘 한 거?** 정책 문서를 파일로 받아서 `ec2:DescribeRouteTables`가 **없는 것**을 확인하고, JSON에 그 권한을 추가했습니다.

### 새 정책 버전 생성

```bash
aws iam create-policy-version --policy-arn <정책ARN> \
  --policy-document file://policy-doc.json --set-as-default
```

- **무슨 명령어?** IAM 정책의 새 버전을 만들고 기본(default) 버전으로 지정합니다. Role에 다시 attach할 필요 없이 즉시 적용됩니다.
- **뭘 한 거?** `ec2:DescribeRouteTables`가 추가된 v2 버전을 만들어 권한 문제를 해결했습니다.

### 컨트롤러 재시작

```bash
kubectl rollout restart deployment/aws-load-balancer-controller -n kube-system
kubectl rollout status deployment/aws-load-balancer-controller -n kube-system
```

- **무슨 명령어?** `rollout restart`는 Deployment의 Pod를 순차 재생성하고, `rollout status`는 재시작이 끝날 때까지 진행 상황을 보여줍니다.
- **뭘 한 거?** 컨트롤러가 백오프(재시도 대기) 상태였기 때문에 재시작으로 즉시 재시도하게 했고, 그 결과 ALB가 생성됐습니다.

## 3. ALB 타겟 헬스체크 진단·수정

### ALB 목록 확인

```bash
aws elbv2 describe-load-balancers \
  --query "LoadBalancers[].{Name:LoadBalancerName,DNS:DNSName,State:State.Code}" --output table
```

- **무슨 명령어?** 계정의 모든 ALB/NLB와 DNS 주소, 상태를 보여줍니다.
- **뭘 한 거?** Terraform이 만든 `forenshield-alb`와 Ingress가 새로 만든 `k8s-forenshi-...` **두 개의 ALB**가 있음을 확인했습니다.

### 타겟 그룹과 타겟 상태 확인

```bash
aws elbv2 describe-target-groups --query "..."
aws elbv2 describe-target-health --target-group-arn <TG_ARN>
```

- **무슨 명령어?** `describe-target-groups`는 ALB에 연결된 타겟 그룹(백엔드/프론트 Pod 묶음)을, `describe-target-health`는 각 타겟(Pod IP)의 헬스 상태를 보여줍니다.
- **뭘 한 거?** 모든 타겟이 `unhealthy / Target.ResponseCodeMismatch`인 것을 발견 → 헬스체크 경로가 `/`라서 백엔드(404)·프론트(307) 모두 실패하는 것을 확인했습니다.

### 헬스체크 경로 변경 후 적용

```bash
kubectl apply -f Infra/config/k8s/backend/service.yaml -f Infra/config/k8s/frontend/service.yaml
```

- **무슨 명령어?** 수정한 yaml을 클러스터에 적용합니다.
- **뭘 한 거?** 각 Service에 `alb.ingress.kubernetes.io/healthcheck-path` 어노테이션(백엔드 `/actuator/health`, 프론트 `/health`)을 추가해 적용 → 전부 healthy로 전환됐습니다.

## 4. Route 53 레코드 전환

### 호스팅 존과 레코드 확인

```bash
aws route53 list-hosted-zones \
  --query "HostedZones[?Name=='forensheildjangdochi.com.'].Id" --output text

aws route53 list-resource-record-sets --hosted-zone-id <존ID> \
  --query "ResourceRecordSets[?Type=='A' || Type=='CNAME']" --output table
```

- **무슨 명령어?** 도메인의 호스팅 존 ID와, 그 안의 DNS 레코드들을 조회합니다.
- **뭘 한 거?** A 레코드가 **옛 ALB**(`forenshield-alb`)를 가리키고 있는 것을 확인 → 503의 직접 원인이었습니다.

### 새 ALB 정보 확인 + 레코드 변경

```bash
aws elbv2 describe-load-balancers --names k8s-forenshi-forenshi-fabf0c2de3 \
  --query "LoadBalancers[0].{DNS:DNSName,Zone:CanonicalHostedZoneId}"

aws route53 change-resource-record-sets --hosted-zone-id Z091888114TC5UULOEOX5 \
  --change-batch file://r53-change.json
```

- **무슨 명령어?** `change-resource-record-sets`는 DNS 레코드를 생성/수정(UPSERT)/삭제합니다. 변경 내용은 JSON 파일로 전달합니다.
- **뭘 한 거?** `forensheildjangdochi.com` A 레코드(Alias)를 새 ALB로 UPSERT → 도메인 접속이 정상화됐습니다.

### 검증

```bash
curl -s -o /dev/null -w "%{http_code}" -L https://forensheildjangdochi.com/   # → 200
curl https://forensheildjangdochi.com/actuator/health                          # → {"status":"UP"}
```

- **무슨 명령어?** `curl -w "%{http_code}"`는 HTTP 상태 코드만 출력하고, `-L`은 리다이렉트를 따라갑니다.
- **뭘 한 거?** 프론트엔드 200, 백엔드 health UP을 확인했습니다.

## 5. 파일 업로드 테스트 + S3 자격증명 문제

### 업로드 테스트

```bash
# 파일명에 공백·괄호가 있으면 따옴표로 감싸고 공백까지 정확히!
curl -F "file=@/c/Users/user/Downloads/aws_architecture.drawio (3).png" \
  https://forensheildjangdochi.com/api/evidences/upload
```

- **무슨 명령어?** `curl -F`는 multipart/form-data로 파일을 업로드합니다. `@경로`가 로컬 파일 지정입니다.
- **뭘 한 거?** 처음엔 `curl (26)` 에러(로컬 파일 못 찾음 — 경로의 공백 누락) → 경로 수정 후 요청은 서버에 도달했지만 `FILE_UPLOAD_FAILED` 응답을 받았습니다.

### 백엔드 로그 확인

```bash
kubectl logs -n forenshield deployment/backend --since=10m
```

- **무슨 명령어?** Deployment에 속한 Pod의 최근 10분치 로그를 출력합니다.
- **뭘 한 거?** `SdkClientException: Unable to load credentials ... the 'sts' service module must be on the class path` 에러를 발견 → AWS SDK `sts` 모듈 누락이 원인임을 확인했습니다.

### Pod의 IRSA 환경변수 확인

```bash
kubectl get pod -n forenshield -l app=backend -o jsonpath="{.items[0].spec.serviceAccountName}"
kubectl exec -n forenshield deployment/backend -- printenv | grep AWS_
```

- **무슨 명령어?** `jsonpath`로 Pod의 ServiceAccount 이름을 뽑고, `exec ... printenv`로 컨테이너 안의 환경변수를 확인합니다.
- **뭘 한 거?** `AWS_ROLE_ARN`, `AWS_WEB_IDENTITY_TOKEN_FILE`이 정상 주입돼 있음을 확인 → IRSA 설정은 문제없고, 순수하게 SDK 의존성 문제로 좁혔습니다.

### 백엔드 재빌드·재배포

```bash
cd /c/Final_Project/backend/backend-forensic

# build.gradle에 implementation 'software.amazon.awssdk:sts' 추가 후
docker build -t forenshield-backend .

aws ecr get-login-password --region ap-northeast-2 | \
  docker login --username AWS --password-stdin 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com

docker tag forenshield-backend:latest 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com/forenshield-backend:latest
docker push 877044078824.dkr.ecr.ap-northeast-2.amazonaws.com/forenshield-backend:latest

kubectl rollout restart deployment/backend -n forenshield
kubectl rollout status deployment/backend -n forenshield
```

- **무슨 명령어?**
  - `docker build`: Dockerfile로 이미지 빌드
  - `aws ecr get-login-password | docker login`: ECR(이미지 저장소) 로그인
  - `docker tag` / `docker push`: ECR 주소로 태그를 붙여 업로드
  - `kubectl rollout restart`: 새 이미지(`imagePullPolicy: Always`)로 Pod 재생성
- **뭘 한 거?** `sts` 의존성이 포함된 새 이미지를 배포 → S3 업로드가 성공했습니다.

## 6. Argo CD 설치·설정

### 설치

```bash
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

- **무슨 명령어?** Argo CD 공식 설치 매니페스트를 `argocd` 네임스페이스에 적용합니다.
- **뭘 한 거?** 설치 도중 `applicationsets.argoproj.io` CRD가 **256KB 어노테이션 제한** 때문에 실패했습니다.

### server-side apply로 재설치

```bash
kubectl apply --server-side --force-conflicts -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

- **무슨 명령어?**
  - `--server-side`: 적용 로직을 API 서버가 처리 → `last-applied-configuration` 어노테이션을 안 만들어서 큰 CRD도 적용 가능
  - `--force-conflicts`: 첫 설치(client-side)와의 필드 소유권 충돌을 강제로 덮어씀
- **뭘 한 거?** 실패했던 CRD까지 전부 적용 완료, 7개 Pod 모두 Running 확인.

### Pod 상태 확인

```bash
kubectl get pods -n argocd
```

- **무슨 명령어?** `argocd` 네임스페이스의 Pod 목록·상태를 확인합니다.
- **뭘 한 거?** server, repo-server, application-controller 등 7개가 전부 `Running`인 것을 확인했습니다.

### 초기 비밀번호 확인

```bash
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d; echo
```

- **무슨 명령어?** Secret의 값은 base64로 인코딩돼 있어서 `base64 -d`로 디코딩해야 실제 비밀번호가 나옵니다.
- **뭘 한 거?** 초기 admin 비밀번호를 확인했지만 로그인이 계속 실패했습니다.

### admin 비밀번호 강제 리셋

```bash
kubectl -n argocd patch secret argocd-secret -p \
  '{"stringData": {"admin.password": "$2a$10$rRyBsGSHK6.uc8fntPwVIuLVHgsAhAX7TcdrqW/RADU0uh7CaChLa", "admin.passwordMtime": "2026-06-10T05:16:00Z"}}'
```

- **무슨 명령어?** `kubectl patch`는 리소스의 일부 필드만 수정합니다. Argo CD는 `argocd-secret`의 `admin.password`(bcrypt 해시)를 로그인 검증에 사용합니다. 위 해시는 `password`라는 문자열의 bcrypt 값입니다.
- **뭘 한 거?** 비밀번호를 `password`로 리셋해서 로그인 문제를 해결했습니다. **로그인 후 UI에서 즉시 변경 필요!**

### UI 접속 (포트포워딩)

```bash
kubectl port-forward svc/argocd-server -n argocd 8443:443
# 브라우저: https://localhost:8443  (admin / 리셋한 비밀번호)
```

- **무슨 명령어?** 클러스터 내부 Service를 내 PC의 포트로 터널링합니다. 외부 노출 없이 안전하게 UI에 접속할 수 있습니다.
- **뭘 한 거?** Argo CD UI에 로그인했습니다.

### Application 등록

```bash
kubectl apply -f Infra/config/argocd/argocd-app.yaml
```

- **무슨 명령어?** Argo CD의 `Application` 커스텀 리소스를 등록합니다. "이 Git 저장소의 이 경로를 이 클러스터/네임스페이스와 동기화해라"라는 선언입니다.
- **뭘 한 거?** `forenshield` 앱을 등록했지만, `repoURL`의 저장소가 GitHub에 아직 없어서 `ComparisonError`가 떴습니다.

### 저장소 존재 확인

```bash
git ls-remote https://github.com/sk-final-deepfake/infra.git
# → remote: Repository not found.
```

- **무슨 명령어?** 원격 저장소의 브랜치/태그 목록을 조회합니다. 저장소 존재 여부 확인에 유용합니다.
- **뭘 한 거?** 저장소가 아직 만들어지지 않았음을 확인 → Infra 폴더 푸시가 선행 작업임을 정리했습니다.
