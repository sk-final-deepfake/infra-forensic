# ForenShield 인프라 및 배포 구조 발표 대본

안녕하세요.

ForenShield의 인프라 및 배포 구조를 발표하겠습니다. ForenShield는 디지털 포렌식 기반 딥페이크 분석 서비스로, AWS EKS 기반의 클라우드 환경과 온프레미스 GPU 서버를 결합한 하이브리드 아키텍처를 사용합니다.
웹 서비스 운영과 데이터 저장은 AWS에서 수행하고, 딥페이크 추론과 같은 고성능 AI 연산은 RTX 5080 GPU 서버에서 처리하도록 설계했습니다.

---

## 1. 서비스 이용 흐름

## 먼저 사용자가 서비스를 이용하는 과정을 설명드리겠습니다. 사용자는 브라우저에서 ForenShield 웹사이트에 접속합니다. 도메인은 Route 53이 관리하며, 사용자의 요청은 ALB를 통해 EKS 클러스터로 전달됩니다.
또한 ACM 인증서를 적용하여 HTTPS 통신을 제공합니다. 웹 화면은 Next.js 기반 프론트엔드가 담당하고 있으며, 사용자는 로그인 후 이미지, 영상, 음성 파일을 업로드할 수 있습니다.
업로드된 파일은 Spring Boot 백엔드로 전달됩니다. 백엔드는 파일에 대해 SHA-256 해시를 생성하고 증거 무결성을 확인하기 위한 Chain of Custody 로그를 기록합니다. 이후 원본 파일은 S3 Evidence 버킷에 저장되고, 파일 정보와 메타데이터는 PostgreSQL 데이터베이스에 기록됩니다.

## 2. 분석 요청 처리 방식

## ForenShield는 분석 요청을 비동기 구조로 처리합니다. 사용자가 분석을 요청하면 Backend는 RabbitMQ에 분석 작업 메시지를 발행합니다. 이 방식은 사용자가 긴 분석 시간 동안 기다리지 않아도 되도록 하기 위해 사용했습니다.
RabbitMQ에 저장된 작업은 AI FastAPI Pod가 가져가 처리합니다. FastAPI는 실제 딥페이크 추론을 수행하는 것이 아니라 작업을 관리하고 GPU 서버에 요청을 전달하는 역할을 수행합니다. 실제 AI 모델 추론은 학교 또는 연구실에 위치한 RTX 5080 GPU 서버에서 실행됩니다. 분석이 완료되면 결과는 PostgreSQL 데이터베이스에 저장되고, 사용자는 웹 화면에서 진행 상태와 결과를 확인할 수 있습니다. 최종적으로 PDF 형태의 분석 리포트도 제공할 예정입니다.

## 3. 사용자 여정 요약

전체 흐름을 9단계로 정리하면 다음과 같습니다.

첫 번째, 사용자가 파일을 업로드합니다.
두 번째, 프론트엔드가 백엔드 API로 데이터를 전달합니다.
세 번째, 백엔드가 SHA-256 해시를 생성하고 CoC 로그를 기록합니다.
네 번째, 파일을 S3에 저장합니다.
다섯 번째, RabbitMQ에 분석 요청 메시지를 등록합니다.
여섯 번째, AI FastAPI Pod가 메시지를 수신합니다.
일곱 번째, 온프레미스 GPU 서버에서 실제 딥페이크 추론을 수행합니다.
여덟 번째, 결과를 PostgreSQL에 저장합니다.
아홉 번째, 사용자가 결과 화면과 보고서를 확인합니다.

## 즉, 사용자는 단순히 웹 서비스만 사용하지만 내부적으로는 API, 메시지 큐, AI 서버, GPU 서버가 연동되어 동작합니다.-

## 4. 인프라 아키텍처

이제 아키텍처를 구역별로 설명드리겠습니다.

### 인터넷 진입 구간

인터넷에서 들어오는 모든 요청은 Route 53을 거쳐 ALB로 전달됩니다.
ALB는 HTTPS 요청을 수신하고 EKS 내부 서비스로 트래픽을 분산합니다.
인터넷과 연결되는 구성요소는 Public Subnet에만 배치하여 보안을 강화했습니다.

---

### EKS 클러스터

## ForenShield의 핵심 애플리케이션은 AWS EKS에서 실행됩니다. EKS는 Kubernetes 기반의 관리형 컨테이너 플랫폼으로, 각 서비스는 Pod 단위로 배포됩니다. Frontend Pod는 Next.js와 Nginx 기반으로 사용자 화면을 제공합니다.
Backend Pod는 Spring Boot 기반으로 회원 관리, 사건 관리, 업로드 처리, 분석 요청 기능을 제공합니다. RabbitMQ Pod는 비동기 분석 작업을 관리합니다. AI FastAPI Pod는 분석 요청을 처리하고 GPU 서버와 통신합니다.
모든 Pod는 Private Subnet에 위치하여 외부에서 직접 접근할 수 없도록 구성했습니다. 또한 Frontend, Backend, AI 서비스를 각각 별도의 Node Group으로 구성하여 독립적인 확장과 배포가 가능하도록 설계했습니다.

### 데이터 저장 계층

데이터 저장은 크게 세 가지 서비스로 구성됩니다.
첫 번째는 RDS PostgreSQL입니다. 사용자 정보, 사건 정보, 분석 결과, Chain of Custody 로그를 저장합니다.
두 번째는 ElastiCache Redis입니다. 세션 데이터와 상태 조회를 캐싱하여 응답 속도를 향상시킵니다.
세 번째는 Amazon S3입니다. 영상, 이미지, 음성과 같은 원본 증거 파일과 AI 모델 파일을 저장합니다.
Evidence 버킷은 버전 관리와 Object Lock 적용을 고려하여 원본 무결성을 보장하도록 설계했습니다.
또한 VPC Endpoint를 사용하여 EKS와 S3 간 통신 시 NAT Gateway를 거치지 않도록 최적화했습니다.

---

### 온프레미스 GPU 서버

## 실제 딥페이크 분석은 온프레미스 GPU 서버에서 수행됩니다.
GPU AI Gateway는 FastAPI 기반으로 구성되어 있으며, EKS에서 전달한 요청을 수신합니다.
이후 RTX 5080 GPU 서버에서 PyTorch 기반 모델을 실행하여 추론을 수행합니다.
클라우드 GPU 대신 온프레미스 GPU를 선택한 이유는 크게 세 가지입니다.
첫째, GPU 비용 절감.
둘째, 연구 데이터와 모델의 직접 관리.
셋째, 대용량 추론 작업에 대한 자원 통제 때문입니다.
현재는 Site-to-Site VPN 대신 EKS NAT Gateway 공인 IP만 GPU 서버 방화벽에서 허용하는 방식으로 운영하고 있습니다.

## 5. 배포 및 운영 구조

## 배포는 GitOps 방식으로 자동화했습니다.
개발자가 GitHub에 코드를 Push하면 GitHub Actions가 실행됩니다.
Actions는 Docker 이미지를 빌드한 후 Amazon ECR에 업로드합니다.
이후 infra-forensic 저장소의 Deployment YAML에 새로운 이미지 태그를 반영합니다.
Argo CD는 Git 저장소를 지속적으로 감시하다가 변경 사항을 감지하면 자동으로 EKS에 적용합니다.
그 결과 새로운 Pod가 Rolling Update 방식으로 배포됩니다.
이를 통해 운영자가 직접 서버에 접속하지 않아도 안정적으로 배포가 가능합니다.
또한 GitHub OIDC와 IAM Role을 사용하여 장기 Access Key 없이 안전하게 AWS 리소스에 접근하도록 구성했습니다.

## 6. 보안 설계

## ForenShield는 포렌식 서비스를 목표로 하기 때문에 보안을 중요하게 고려했습니다.
모든 애플리케이션과 데이터베이스는 Private Subnet에 배치했습니다.
ALB를 제외한 구성요소는 인터넷에 직접 노출되지 않습니다.
통신은 HTTPS를 사용하며 ACM 인증서를 적용했습니다.
업로드되는 모든 파일은 SHA-256 해시와 Chain of Custody 로그를 생성하여 무결성을 검증합니다.
데이터베이스와 RabbitMQ 접속 정보는 Kubernetes Secret으로 관리하며 Git 저장소에는 포함하지 않습니다.
또한 GPU 서버는 지정된 NAT Gateway 공인 IP만 접근할 수 있도록 방화벽 규칙을 적용했습니다.

## 7. 마무리

정리하겠습니다.
ForenShield는 AWS EKS 기반의 클라우드 환경과 온프레미스 GPU 서버를 결합한 하이브리드 아키텍처를 사용합니다.
웹 서비스와 데이터 저장, 자동 배포는 AWS에서 담당하고, 고성능 딥페이크 추론은 RTX 5080 GPU 서버에서 수행합니다.
또한 GitHub Actions와 Argo CD 기반의 GitOps 환경을 구축하여 안정적인 자동 배포 체계를 구현했습니다.
향후에는 VPN 기반 보안 강화, 모델 자동 배포, PDF 리포트 고도화, 멀티모달 딥페이크 분석 기능을 추가할 계획입니다.
이상으로 ForenShield 인프라 및 배포 구조 발표를 마치겠습니다.

감사합니다.
