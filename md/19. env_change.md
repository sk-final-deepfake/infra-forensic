## Step 1 — RDS 비밀번호 교체

**대상:** `POSTGRES_PASSWORD` (마스터 사용자 `forenshield`)

### AWS 콘솔

1. [RDS 콘솔](https://ap-northeast-2.console.aws.amazon.com/rds/home?region=ap-northeast-2) 이동
2. **Databases** → `forenshield-db` 선택
3. **Modify** 클릭
4. **Credentials management** → **Master password** → **Self managed**
5. **New master password** / **Confirm password**에 `<NEW_RDS_PASSWORD>` 입력
6. **Continue** → 변경 시점 **Apply immediately** 선택 → **Modify DB instance**
7. 상태가 **Available**이 될 때까지 대기 (수 분)

### AWS CLI

```bash
aws rds modify-db-instance \
  --db-instance-identifier forenshield-db \
  --master-user-password "<NEW_RDS_PASSWORD>" \
  --apply-immediately

aws rds wait db-instance-available \
  --db-instance-identifier forenshield-db
```

### 확인

```bash
aws rds describe-db-instances \
  --db-instance-identifier forenshield-db \
  --query 'DBInstances[0].DBInstanceStatus' \
  --output text
# 기대: available
```

> **주의:** 이 시점부터 EKS Pod는 **아직 옛 비밀번호**를 쓰므로 DB 연결 오류가 날 수 있습니다. Step 6~7을 빠르게 이어서 진행하세요.

## Step 2 — ElastiCache Redis AUTH 교체

**대상:** `REDIS_PASSWORD` (AUTH token)  
**엔드포인트:** `master.forenshield-redis.yvtd5s.apn2.cache.amazonaws.com`

### AWS 콘솔

1. [ElastiCache 콘솔](https://ap-northeast-2.console.aws.amazon.com/elasticache/home?region=ap-northeast-2) → **Redis OSS caches**
2. `forenshield-redis` 선택
3. **Modify** 클릭
4. **Access control** → **AUTH default user access** → **AUTH token**
5. **AUTH token**에 `<NEW_REDIS_AUTH>` 입력
6. **Update strategy** → **Rotate** (가능하면 선택 — 전환 중 이중 토큰 허용)
7. **Preview changes** → **Modify**

### AWS CLI

```bash
aws elasticache modify-replication-group \
  --replication-group-id forenshield-redis \
  --auth-token "foren123456Secure!" \
  --auth-token-update-strategy ROTATE \
  --apply-immediately

aws elasticache wait replication-group-available \
  --replication-group-id forenshield-redis
```

`ROTATE`가 실패하면 `SET`(즉시 교체)로 시도합니다. 이 경우 Step 6~7을 **연속으로** 진행해 다운타임을 최소화하세요.

```bash
aws elasticache modify-replication-group \
  --replication-group-id forenshield-redis \
  --auth-token "<NEW_REDIS_AUTH>" \
  --auth-token-update-strategy SET \
  --apply-immediately
```

### 확인

```bash
aws elasticache describe-replication-groups \
  --replication-group-id forenshield-redis \
  --query 'ReplicationGroups[0].Status' \
  --output text
# 기대: available
```

## Step 4 — RabbitMQ 비밀번호 교체

**대상:** `RABBITMQ_PASSWORD`  
**배포:** EKS `forenshield` 네임스페이스, Helm release `rabbitmq` (Bitnami)

RabbitMQ는 AWS 관리형이 아니라 **EKS 안**에 있으므로 AWS 콘솔 대신 **EKS / kubectl / Helm**으로 작업합니다.

### 방법 A — Helm upgrade (권장)

#### 준비 — values 파일 (Git에 비밀번호 넣지 말 것)

로컬에만 `rabbitmq-values.local.yaml` 생성:

```yaml
auth:
  username: forenshield
  password: "<NEW_RABBITMQ_PASSWORD>"
```

#### CLI

```bash
helm upgrade rabbitmq bitnami/rabbitmq \
  -n forenshield \
  --version 14.6.9 \
  -f rabbitmq-values.local.yaml \
  --reuse-values

kubectl delete pod rabbitmq-0 -n forenshield
kubectl wait --for=condition=Ready pod/rabbitmq-0 -n forenshield --timeout=300s
```

#### AWS 콘솔 (EKS에서 상태만 확인)

1. [EKS 콘솔](https://ap-northeast-2.console.aws.amazon.com/eks/home?region=ap-northeast-2) → `forenshield` → **Resources** 탭
2. **Workloads** → Namespace `forenshield` → Pod `rabbitmq-0` → **Running** 확인

### 방법 B — K8s Secret만 갱신 (Helm과 불일치 주의)

Helm이 Secret을 덮어쓸 수 있으므로 **방법 A를 우선**하세요.  
긴급 시 Step 6 `apply-settings.sh`의 `rabbitmq-credentials` Secret 갱신 + Pod 재시작으로 backend/ai-fastapi 연결만 복구할 수 있습니다.

### 확인

```bash
kubectl get pods -n forenshield -l app.kubernetes.io/name=rabbitmq
kubectl logs rabbitmq-0 -n forenshield --tail=20
```

## Step 5 — GitHub PAT 교체 (해당 시)

`.env` 또는 커밋 로그에 `ghp_` 로 시작하는 **Personal Access Token**이 있었다면 즉시 폐기합니다.

### GitHub 웹 (콘솔)

1. GitHub → **Settings** → **Developer settings** → **Personal access tokens**
2. 노출된 토큰 **Delete / Revoke**
3. 새 토큰 발급 — Classic: **`repo`** scope (private `infra-forensic` push용)
4. 조직 SSO 사용 시 토큰 **Authorize**

### GitHub CLI

```bash
# 새 토큰은 웹에서 발급 후 아래 Secret에 등록
gh secret set INFRA_REPO_TOKEN --repo sk-final-deepfake/frontend-deepfake --body "<NEW_PAT>"
gh secret set INFRA_REPO_TOKEN --repo sk-final-deepfake/backend-forensic --body "<NEW_PAT>"
gh secret set INFRA_REPO_TOKEN --repo sk-final-deepfake/ai-forensic --body "<NEW_PAT>"
```

| 저장소 | Secret 이름 |
|--------|-------------|
| `frontend-deepfake` | `INFRA_REPO_TOKEN` |
| `backend-forensic` | `INFRA_REPO_TOKEN` |
| `ai-forensic` | `INFRA_REPO_TOKEN` |

### AWS Access Key (해당 시)

커밋된 `.env`에 `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`가 있었다면:

#### AWS 콘솔

1. [IAM 콘솔](https://console.aws.amazon.com/iam/) → **Users** → 해당 사용자
2. **Security credentials** → 노출된 Access Key **Deactivate** → **Delete**
3. 필요 시 새 Access Key 발급 (운영은 **IRSA/OIDC** 권장)

#### AWS CLI

```bash
aws iam list-access-keys --user-name <IAM_USER_NAME>
aws iam update-access-key --user-name <IAM_USER_NAME> --access-key-id <KEY_ID> --status Inactive
aws iam delete-access-key --user-name <IAM_USER_NAME> --access-key-id <KEY_ID>
```

## Step 9 — Git 저장소 정리

> **전제:** Step 1~8 **완료 후** 진행. 비밀번호를 바꾸기 전에 히스토리만 지워도 **이미 유출된 값**은 위험합니다.

### 9.1 추적 중인 민감 파일 점검

```bash
# 각 저장소에서
git ls-files | grep -E '\.env$|secrets\.env|rabbitmq-values'
```

**현재 알려진 노출 지점:**

| 저장소 | 파일 | 조치 |
|--------|------|------|
| `backend-forensic` | `.env` (과거 커밋) | 히스토리 purge |
| `infra-forensic` | `.env.example` (실값 포함) | placeholder로 수정 후 push |
| `infra-forensic` | `tmp/rabbitmq-values.yaml` | 비밀번호 제거 또는 repo에서 삭제 |

### 9.2 `.env.example` sanitize (즉시 push 가능)

실제 비밀번호·JWT를 **빈 값**으로:

```bash
POSTGRES_PASSWORD=
REDIS_PASSWORD=
RABBITMQ_PASSWORD=
JWT_SECRET_KEY=
```

호스트명·버킷명·계정 ID는 비밀이 아니므로 유지 가능.

### 9.3 Git 히스토리 purge (CLI)

팀 전원에게 **force push · fresh clone** 공지 후 진행.

```bash
# git-filter-repo 설치 필요: pip install git-filter-repo

cd backend-forensic
git filter-repo --path .env --invert-paths --force
git push origin --force --all
git push origin --force --tags
```

`infra-forensic`도 `.env.example` 옛 버전·`tmp/rabbitmq-values.yaml`에 대해 동일하게 적용합니다.

### 9.4 GitHub 웹에서 확인

1. 저장소 → **Commits** / **Blame**으로 `.env` 검색
2. GitHub **Secret scanning** 알림이 있으면 **모두 Revoke 완료** 처리
3. Private repo라도 **협업자·fork·캐시** 대비해 purge 권장

## 부록 — 노출되어도 교체 불필요한 항목

| 항목 | 이유 |
|------|------|
| `AWS_ACCOUNT_ID`, `ECR_REGISTRY` | 공개 정보에 가까움 |
| `POSTGRES_HOST`, `REDIS_HOST` | 엔드포인트 (SG로 보호) |
| `S3_*_BUCKET` 버킷명 | IAM/IRSA 없이는 접근 불가 |
| `AI_GATEWAY_URL`, `NEXT_PUBLIC_API_URL` | URL (비밀 아님) |
| `AWS_ROLE_ARN` (IRSA) | Role ARN 자체는 비밀이 아님 |
