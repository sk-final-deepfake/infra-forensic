## 1. 로컬 개발 — API 401 · mock 데이터

### 1-1. 증상

- 로컬 프론트(`localhost:3000`)에서 관리자 API가 500 → 401 연쇄
- Guard는 통과하는데 API만 실패

### 1-2. 원인

1. **`.env`** 에 `NEXT_PUBLIC_API_URL=https://forensheildjangdochi.com` (배포 URL)
2. 브라우저 `sessionStorage`에 **배포 서버 JWT** 잔존 → 로컬 백엔드가 거부
3. `NEXT_PUBLIC_USE_MOCK_API` 미설정 시 `lib/api/admin.ts`가 **mock fallback** 사용

```ts
// lib/api/admin.ts
const USE_MOCK_ADMIN = process.env.NEXT_PUBLIC_USE_MOCK_API !== "false"
```

### 1-3. 해결

**`frontend-deepfake/.env.local`** (`.env`보다 우선):

```env
BACKEND_API_ORIGIN=http://localhost:8080
NEXT_PUBLIC_API_URL=http://localhost:8080
NEXT_PUBLIC_USE_MOCK_API=false
```

추가 조치:

1. `npm run dev` **재시작**
2. 브라우저 **로그아웃** 또는 시크릿 창
3. 로컬 백엔드 `localhost:8080` 실행
4. 관리자 시드: `3333` / `4444`


```bash
git pull origin develop
```

## 5. 롤아웃 멈춤 — Too many pods

### 5-1. 증상

```bash
kubectl rollout restart deployment/frontend -n forenshield
kubectl rollout status deployment/frontend -n forenshield
```

```text
Waiting for deployment "frontend" rollout to finish: 1 old replicas are pending termination...
```

### 5-2. Pod 상태

```text
frontend-5c79797bff-xxx   2/2   Running   (구 Pod, 23h)
frontend-7b6d56df7-xxx    0/2   Pending   (신 Pod)
```

### 5-3. 원인

`kubectl describe pod` (신 Pod):

```text
Warning  FailedScheduling  0/4 nodes are available:
  1 Too many pods,
  3 node(s) didn't match Pod's node affinity/selector
```

- frontend Deployment: `nodeSelector: nodegroup=frontend-ng`
- **`frontend-ng` 노드 1대** (`t3.medium`) — Pod **17/17 한도**
- Rolling Update 시 잠깐 Pod 2개 필요 → **18번째 슬롯 없음** → 신 Pod `Pending`
- 구 Pod는 신 Pod가 Ready 될 때까지 종료 안 됨 → **데드락**

**`frontend-ng` 노드 Pod 구성 (당시):**

| 네임스페이스 | Pod 수 | 비고 |
|-------------|--------|------|
| `argocd` | 7 | Argo CD 전체가 frontend 노드에 배치됨 |
| `kube-system` | 9 | CoreDNS, ALB controller, EBS CSI 등 |
| `forenshield` | 1 | frontend (구) |
| **합계** | **17/17** | |

### 5-4. 해결 (당장)

**방법 A — 구 Pod 삭제 (가장 빠름, 수 초 다운타임)**

```bash
kubectl delete pod <구-frontend-pod-이름> -n forenshield
kubectl rollout status deployment/frontend -n forenshield
```

**방법 B — `frontend-ng` 노드 1대 추가**

```bash
# AWS EKS 콘솔 또는 eksctl
# frontend-ng desired capacity: 1 → 2
```

**방법 C — Recreate 전략 (짧은 중단 허용 시)**

```bash
kubectl patch deployment frontend -n forenshield \
  -p '{"spec":{"strategy":{"type":"Recreate"}}}'
kubectl rollout restart deployment/frontend -n forenshield
```

### 5-5. 근본 대책 (권장)

1. **Argo CD를 `frontend-ng`가 아닌 노드그룹으로 이전** (7 Pod 점유 해소)
2. `frontend-ng` **최소 2노드** 또는 롤아웃 전략 `Recreate` 검토
3. 배포 후 `kubectl get pods -n forenshield -l app=frontend` 로 **2/2 Running** 확인

## 6. 배포 설정에서 빠진 것

### 6-1. `NEXT_PUBLIC_USE_MOCK_API=false` (Docker 빌드)

| 위치 | 현재 |
|------|------|
| `Dockerfile` | `NEXT_PUBLIC_API_URL` 만 `ARG` |
| `deploy-develop.yml` | 동일 |
| **빠진 것** | `NEXT_PUBLIC_USE_MOCK_API=false` |

배포 이미지에서 env 미설정 → `!== "false"` 조건으로 **mock 모드 기본 ON** → API 실패 시 mock 데이터 표시.

**권장 수정:**

```dockerfile
ARG NEXT_PUBLIC_USE_MOCK_API=false
ENV NEXT_PUBLIC_USE_MOCK_API=$NEXT_PUBLIC_USE_MOCK_API
```

```yaml
# deploy-develop.yml build step
--build-arg NEXT_PUBLIC_USE_MOCK_API=false
```

### 6-2. `frontend-config` ConfigMap

```yaml
# config/k8s/frontend-config.yaml
NEXT_PUBLIC_API_URL: https://forensheildjangdochi.com/api   # ← /api 접미사
```

Next.js `NEXT_PUBLIC_*` 는 **빌드 타임**에 번들에 박힘.  
런타임 ConfigMap만 바꿔도 **클라이언트 API URL은 안 바뀜**.  
CI Secret `NEXT_PUBLIC_API_URL` 은 `https://forensheildjangdochi.com` (**/api 없이**) 이어야 함 ([13번 문서](./13.frontend-cicd-argocd-deploy.md) §8).

### 6-3. 알려진 API 갭 (UI 근사)

| UI 항목 | 상태 |
|---------|------|
| 소속기관 수 | 전용 API 없음 → logs `departments` 길이로 근사 |
| 승인 처리 이력 | 별도 API 없음 → users + `joinedAt` |
| 계정 정지 | suspend API 없음 → 버튼 비활성 |

| Tunnel 2 Outside (AWS) | `43.203.31.222` |
| Tunnel 2 Inside (On-Prem) | `169.254.251.150/30` |

### 8-2. AWS 콘솔 작업 순서 (당일 진행)

1. **Customer Gateway** `forenshield-cgw` — On-Prem 공인 IP
2. **VGW** `forenshield-vgw` — VPC attach
3. **Site-to-Site VPN** — Static routing, `192.168.0.0/24`
4. **VPN Configuration File** 다운로드 (Vendor: Generic) → GPU 서버에 전달
5. **Route Table** — Private RT에 VGW **Route propagation** 활성화
6. **Security Group** — `vpn-test`(또는 VPN용 SG)에 `192.168.0.0/24` 인바운드

### 8-3. GPU 서버 strongSwan (터널 2만)

기존 `/etc/ipsec.conf`에 **Tunnel1** (`right=3.35.116.207` 등 예전 IP)이 있었음 → **삭제 후 Tunnel2로 교체**.

```text
# /etc/ipsec.conf 핵심
conn Tunnel2
    leftid=58.127.241.84
    leftsubnet=192.168.0.0/24
    leftsourceip=169.254.251.150
    right=43.203.31.222          # Tunnel 2 AWS Outside IP
    rightsubnet=10.0.0.0/16
    ike=aes128-sha1-modp1024!
    esp=aes128-sha1-modp1024!
    auto=start
```

`/etc/ipsec.secrets`:

```text
58.127.241.84 43.203.31.222 : PSK "Tunnel2-PSK"
```

### 8-4. 트러블슈팅

| 증상 | 원인 | 해결 |
|------|------|------|
| Tunnel 1 DOWN | 터널 2만 쓰기로 함 | **정상** — Tunnel 2 UP이면 OK |
| `parsing error, unknown keyword 'mtu'` | strongSwan 5.9에서 `mtu=` 미지원 | `mtu` 줄 삭제 → `sudo ipsec restart` |
| `forenshield-sg-vpn` 없음 | 테스트용 `vpn-test` SG 사용 | 이름 무관, **On-Prem CIDR 인바운드**만 확인 |
| VPN UP인데 AI 호출 실패 | `AI_GATEWAY_URL`이 아직 공인 IP | ConfigMap을 GPU **사설 IP**로 변경 + ai-fastapi 재시작 |

### 8-5. 성공 확인 (당일)

```bash
# GPU 서버
sudo ipsec statusall
# Tunnel2[1]: ESTABLISHED
# child: 192.168.0.0/24 === 10.0.0.0/16

# AWS 콘솔 — Site-to-Site VPN → Tunnel 2 = UP
```

## 9. env_change — 비밀번호·Secret 교체 트러블슈팅

> 절차 원본: [`19. env_change.md`](./19.%20env_change.md) (Git 유출 대응 — RDS · Redis · JWT · RabbitMQ)

### 9-1. 배경

`.env` / `rabbitmq-values.yaml` 등에 **실제 비밀번호가 GitHub에 올라간 경우** → AWS·K8s 자격증명 **먼저 교체** 후 `secrets.env` → `apply-settings.sh` → Pod 재시작.

### 9-2. `<NEW_RDS_PASSWORD>` placeholder 실수

**증상:** RDS 비번 바꿨는데 앱 DB 연결 실패.

**원인:** 문서 placeholder `<NEW_RDS_PASSWORD>`를 **문자 그대로** CLI에 넣음.

```bash
# ❌ 잘못 — 글자 그대로 RDS 비밀번호가 됨
--master-user-password "<NEW_RDS_PASSWORD>"

# ✅ 올바름
export NEW_RDS_PASSWORD='실제_새_비밀번호'
aws rds modify-db-instance \
  --db-instance-identifier forenshield-db \
  --master-user-password "$NEW_RDS_PASSWORD" \
  --apply-immediately
```

### 9-3. RabbitMQ Helm — bitnami repo 없음

```text
no cached repo found. (try 'helm repo update')
bitnami-index.yaml: The system cannot find the file specified
```

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
cd /c/Final_Project/Infra
helm upgrade rabbitmq bitnami/rabbitmq \
  -n forenshield --version 14.6.9 \
  -f rabbitmq-values.local.yaml --reuse-values
```

> `-f` 경로: `rabbitmq-values.local.yaml`은 **`Infra/`** 아래 — `cd Infra` 후 실행.

### 9-4. RabbitMQ Ready 안 됨 — probe 401

**증상:**

```text
Readiness probe failed: curl: (22) ... 401
kubectl wait ... --timeout=300s   # 무한 대기처럼 보임
```

**원인:** Helm upgrade는 Secret만 바꿔도 **PVC 안 RabbitMQ 데이터(옛 비번)** 는 그대로.

**해결 (큐 데이터 삭제 가능 시):**

```bash
kubectl scale statefulset rabbitmq -n forenshield --replicas=0
kubectl wait --for=delete pod/rabbitmq-0 -n forenshield --timeout=120s
kubectl delete pvc data-rabbitmq-0 -n forenshield
kubectl scale statefulset rabbitmq -n forenshield --replicas=1
kubectl wait --for=condition=Ready pod/rabbitmq-0 -n forenshield --timeout=300s
```

**PVC `Terminating`에서 멈춤:** Pod가 디스크를 붙잡고 있음 → **STS replicas=0** 먼저.

### 9-5. `apply-settings.sh` — `RDS_ENDPOINT` unbound

**증상:**

```text
config/apply-settings.sh: line 69: RDS_ENDPOINT: unbound variable
error: no objects passed to apply
```

**원인:** `config/infra.env` / `infra.env.example` **없음** — `secrets.env`만 있음.

**해결:** `Infra/config/infra.env` 생성 (비밀 없음):

```bash
S3_EVIDENCE_BUCKET=forenshield-evidence-877044078824
S3_MODELS_BUCKET=forenshield-models-877044078824
CLUSTER_NAME=forenshield
```

```bash
cd /c/Final_Project/Infra

```

**성공 시 출력:**

| Secret | 결과 |
|--------|------|
| `db-credentials` | configured |
| `redis-credentials` | configured |
| `rabbitmq-credentials` | configured |
| `app-secrets` | unchanged (JWT 안 바꿨으면 정상) |
