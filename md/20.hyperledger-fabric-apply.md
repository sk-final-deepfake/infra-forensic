# Hyperledger Fabric 적용 가이드 (ForenShield)

> **작성 기준:** 2026-06-23  
> **범위:** ForenShield / VeriForensics에 Fabric **해시 앵커** 연동 시 INF·BE·FE가 해야 할 일  
> **관련 RQ:** REQ-052 · DTL-078~~080 · SEC-151~~152  

---

## 1. 한 줄 요약

- **BE는 이미 구현됨** — 해시 계산·DB·앵커 트리거·검증 API (`mode=simulated` | `http`)
- **Fabric은 INF가 배포** — 네트워크 + chaincode + **Anchor Gateway(REST)**
- **온체인에는 SHA-256 해시만** — 영상·PDF·PII 미포함
- **BE는 Fabric SDK/지갑을 직접 쓰지 않음** — HTTP 게이트웨이만 호출

상세 BE 구현: `[backend/backend-forensic/docs/architecture/blockchain-implementation.md](../../backend/backend-forensic/docs/architecture/blockchain-implementation.md)`  
HTTP 계약: `[backend/backend-forensic/docs/integrations/blockchain.md](../../backend/backend-forensic/docs/integrations/blockchain.md)`

## 2. 아키텍처

```text
[User] → [FE] → [BE Pod / EKS]
                    │
                    ├─ S3 (증거·PDF)
                    ├─ PostgreSQL (evidence, CoC, blockchain_anchors)
                    │
                    └─ POST {subjectHash, anchorType}
                              │
                              ▼
                    [INF: Fabric Anchor Gateway]  ← 신규 구축
                              │
                              ▼
                    [Hyperledger Fabric]
                      peer · orderer · CA · channel · chaincode
```


| 레이어 | 담당  | 역할                                        |
| --- | --- | ----------------------------------------- |
| BE  | 백엔드 | 업무·해시·CoC·Merkle·`blockchain_anchors`·API |
| INF | 인프라 | Fabric 네트워크·chaincode·Gateway·CA·Explorer |
| FE  | 프론트 | `blockchainInfo`·TX 링크·`hashValid` 표시     |

## 3. Fabric이 하는 일 (우리 프로젝트)


| 앵커 타입           | BE 트리거              | Fabric에 기록되는 값                      |
| --------------- | ------------------- | ----------------------------------- |
| `EVIDENCE_HASH` | 증거 업로드 완료           | 원본 SHA-256                          |
| `REPORT_HASH`   | PDF 리포트 생성          | PDF SHA-256                         |
| `MERKLE_ROOT`   | 매일 01:00 (또는 Admin) | 전일 CoC `currentLogHash` Merkle Root |


**하지 않는 것:** 파일 저장, AI 분석, 로그인, CoC 건별 TX

## 5. INF — 해야 할 일

### 5.1 네트워크 (PoC 최소)


| 구성요소         | PoC                    | 운영 권장             |
| ------------ | ---------------------- | ----------------- |
| Organization | 1 (`OrgForensic`)      | 2+ (기관별)          |
| Channel      | `forenshield-evidence` | 사건/민감도별 추가 가능     |
| Orderer      | Raft 1 node            | Raft 3+ (HA)      |
| Peer         | 1                      | 기관당 1+            |
| CA           | Fabric CA 1            | 운영 CA + 인증서 갱신 절차 |


**배포 형태 예:** EC2 VM 1~2대 + Docker Compose, 또는 EKS 별 namespace

### 5.2 Chaincode (`anchor`)

**함수 (개념)**


| 함수                                              | 용도                     |
| ----------------------------------------------- | ---------------------- |
| `AnchorHash(subjectHash, anchorType, metadata)` | 해시 앵커 기록 (append-only) |
| `GetAnchor(key)` / `GetByHash(subjectHash)`     | 감사·검증 조회               |


**온체인 payload 예**

```json
{
  "subjectHash": "64자 hex SHA-256",
  "anchorType": "EVIDENCE_HASH | REPORT_HASH | MERKLE_ROOT",
  "evidenceId": "optional",
  "reportId": "optional",
  "anchoredAt": "ISO-8601 UTC",
  "clientId": "forenshield-be"
}
```

**정책**

- 동일 `anchorType` + 대상 ID에 대해 **멱등** (BE도 ANCHORED 시 스킵)
- 해시 **덮어쓰기 금지** (Update로 변경 불가)
- endorsement PoC: `OR('OrgForensic.peer')`

### 5.3 Anchor Gateway (필수)

BE와의 **HTTP 계약** (변경 시 BE·문서 동시 수정)

**Request** `POST /api/v1/anchor` (경로는 INF 정의, BE `http-url`에 전체 URL 설정)

```json
{
  "subjectHash": "abc...",
  "anchorType": "EVIDENCE_HASH"
}
```

**Response**

```json
{
  "transactionHash": "fabric-tx-id-or-hex",
  "blockNumber": 12345
}
```

**Gateway 내부**

1. Fabric Gateway SDK로 `AnchorHash` invoke
2. 성공 시 `transactionId` → `transactionHash`, 블록 번호 반환
3. 실패 시 HTTP 5xx + 메시지 (BE는 `FAILED` 저장)

**보안**

- Gateway만 Fabric invoke identity 보유 (**BE에 private key 없음**)
- 내부망/VPN/mTLS 권장
- (선택) API Key 또는 IP allowlist

### 5.4 Explorer (선택)

- TX 조회 UI (Hyperledger Explorer 등)
- BE 설정: `blockchain.anchor.explorer-url-template`  
예: `https://fabric-explorer.internal/transaction/{txHash}`

### 5.5 INF 체크리스트

```text
[ ] VM/EC2 또는 K8s namespace 준비
[ ] Fabric 2.5.x test-network 기동·문서화
[ ] Channel 생성·chaincode install/approve/commit
[ ] Gateway 서비스 배포 + health check
[ ] Gateway → Fabric 연결 테스트 (수동 invoke 1건)
[ ] BE staging에서 http-url 연결
[ ] 로그·백업(peer ledger) 절차
[ ] 일일 비용: 미사용 시 VM 중지 (→ md/14.daily-cost-shutdown.md)
```

## 6. BE — 해야 할 일

**코드 대규모 수정 불필요.** 설정·검증 위주.

### 6.1 설정 (`application-prod.yaml` 또는 K8s ConfigMap)

```yaml
blockchain:
  anchor:
    enabled: true
    mode: http                    # simulated → http
    network: hyperledger-fabric-forenshield
    http-url: https://fabric-gateway.internal/api/v1/anchor
    scheduler-enabled: true
    daily-cron: "0 0 1 * * *"
    explorer-url-template: https://fabric-explorer.internal/transaction/{txHash}
```

**환경 변수:** `BLOCKCHAIN_ANCHOR_MODE`, `BLOCKCHAIN_ANCHOR_URL`, `BLOCKCHAIN_ANCHOR_NETWORK`, `BLOCKCHAIN_EXPLORER_URL_TEMPLATE`

### 6.2 BE 체크리스트

```text
[ ] staging/prod ConfigMap·Secret에 http-url 반영
[ ] mode=http 시 HttpBlockchainAnchorClient 활성 확인
[ ] 증거 업로드 → blockchain_anchors ANCHORED
[ ] PDF 생성 → REPORT_HASH ANCHORED
[ ] Merkle 스케줄 또는 Admin POST → MERKLE_ROOT
[ ] GET .../detail → blockchainInfo.hashValid
[ ] GET .../integrity/verify → BLOCKCHAIN_HASH 항목
[ ] FAILED 시 error_message·모니터링 알림
```

### 6.3 BE가 하지 않는 것

- Fabric peer/orderer 배포
- chaincode 배포
- 지갑·X.509 관리

## 7. FE — 해야 할 일

```text
[ ] 증거 상세 blockchainInfo 표시 (status, hashValid, txHash)
[ ] explorer-url-template 링크 (INF URL 확정 후)
[ ] 앵커 FAILED 시 사용자/운영 안내 (선택)
```

API: `GET /api/v1/evidences/{id}/blockchain`, `GET .../detail` → `blockchainInfo`

## 9. 비용·리소스 (참고)


| 구간    | 비용 성격                   | 대략                                     |
| ----- | ----------------------- | -------------------------------------- |
| PoC   | EC2 VM 1~2 + Docker     | 월 수만~십만 원대 (스펙·리전별)                    |
| 운영 HA | peer/orderer/Gateway 다수 | 월 수십~수백만 원+ + INF 운영 인력                |
